## Finding 1: Phase 3 atomic sed glob misses 6 fail-closed gate wrappers → inbound + recurrence gates deadlock
- **Severity:** Critical
- **Location:** Phase 3, "Implementation Steps" step 2 (sed command); Phase 1 inventory "Cross-runtime refs (Phase 3 atomic)"; plan.md acceptance criterion "All 9 coordination-gate files"
- **Flaw:** The Phase 3 sed uses glob `*-coordination-gate.cjs`, which matches only `bash-coordination-gate.cjs` and `write-coordination-gate.cjs` (6 files across 3 runtimes). The coordination/hooks/ dirs contain TWO MORE fail-closed wrapper files per runtime — `inbound-state-gate.cjs` and `recurrence-check-on-start.cjs` — that also resolve `tools/learning-loop-mastra/hooks/legacy/<gate>.js` via `execFileSync` and `process.exit(err.status ?? 1)` on a missing path. The plan's inventory calls them `{bash,write,inbound,recurrence}-coordination-gate.cjs:13`, but inbound and recurrence wrappers are NOT named `*-coordination-gate.cjs`. The glob silently misses them.
- **Failure scenario:** Phase 3 atomic command runs: `git mv hooks/legacy hooks/universal` succeeds; sed rewrites only bash+write coordination gates. The 6 `inbound-state-gate.cjs` / `recurrence-check-on-start.cjs` files (3 runtimes × 2) still point at `hooks/legacy/` (now moved). Next `UserPromptSubmit` → inbound-state-gate.cjs → `execFileSync` missing path → `process.exit(1)` → inbound gate deadlocks. This is the exact bootstrapping deadlock the plan exists to prevent, induced by the plan's own atomic command. The Phase 3 "immediate gate smoke" only tests bash + write gates (step 4), so it would not catch the inbound/recurrence deadlock until a real prompt arrives.
- **Evidence:** `ls .claude/coordination/hooks/` → `bash-coordination-gate.cjs, inbound-state-gate.cjs, recurrence-check-on-start.cjs, write-coordination-gate.cjs`. `.claude/coordination/hooks/inbound-state-gate.cjs:13` = `const universalHook = path.join(projectRoot, 'tools/learning-loop-mastra/hooks/legacy/inbound-gate.js');` followed by `execFileSync` + `process.exit(err.status ?? 1)`. `.claude/coordination/hooks/recurrence-check-on-start.cjs:13` same pattern for `recurrence-check-on-start.js`. `ls *-coordination-gate.cjs` across 3 runtimes = 6 (plan claims 9). The 6 non-matching wrappers all carry the identical fail-closed hazard.
- **Suggested fix:** Replace the Phase 3 sed glob with an explicit file list (or a glob that matches all 4 wrapper patterns): `sed -i 's#hooks/legacy/#hooks/universal/#g' ../../.claude/coordination/hooks/{bash,write}-coordination-gate.cjs ../../.claude/coordination/hooks/inbound-state-gate.cjs ../../.claude/coordination/hooks/recurrence-check-on-start.cjs <repeat for .factory, .mastracode>`. Add the 6 missed wrappers to the Phase 1 inventory. Correct the "9 coordination-gate files" count to "12 fail-closed wrappers (6 `*-coordination-gate.cjs` + 6 `inbound-state-gate.cjs`/`recurrence-check-on-start.cjs`)". Extend Phase 3 step 4 smoke to trigger inbound + recurrence resolution too.

## Finding 7: Gold-plating — `gate_override` + TTL + audit-note + symlink-fallback machinery is speculative ceremony
- **Severity:** Medium
- **Location:** Phase 1 step 4; Phase 1 Risk Assessment; Phase 3 Risk Assessment ("last resort: symlink")
- **Flaw:** The plan builds a 3-layer escape-hatch apparatus (operator `gate_override` scoped+TTL'd+audit-noted; then a last-resort `hooks/universal/`→`hooks/legacy/` symlink; then reverse-move recovery) for a bootstrapping hazard whose actual gate-block status is never established as having occurred. The plan's own Phase 1 step 4 says "If `ok`, proceed" — i.e., the apparatus is only needed if the gate self-protects, which the plan never shows it does. The atomic command is already a single bash invocation evaluated by the PRE-move gate (the plan's correct core insight), so the deadlock is structurally avoided; the override/symlink machinery is defense-in-depth against a failure mode not demonstrated to exist.
- **Failure scenario:** Unnecessary cost: the plan spends a full Phase-1 step + two Risk Assessments + a documented recovery procedure on a speculative gate-self-protection rule. The simpler approach — just run the atomic command; if the bash-gate blocks it, THEN escalate — needs no pre-built TTL/audit/symlink apparatus. Worse, the symlink fallback (a `hooks/universal/` symlink to `hooks/legacy/` created before the move) re-introduces a transient dual-path that the rename was meant to eliminate, and if forgotten in Phase 4, leaves a stale symlink that masks the rename.
- **Evidence:** Phase 1 step 4 + Risk Assessment describe the override + symlink; Phase 3 Risk Assessment repeats the symlink "last resort." No grep evidence shows the bash-gate actually blocks `git mv` on `hooks/legacy/` or edits to coordination-gate files (the plan never runs `gate_check` and records a `block` result).
- **Suggested fix:** Drop the pre-built override/TTL/symlink apparatus. Phase 1 step 4 becomes: run `gate_check` on the Phase-3 command; if `ok`, proceed; if `block`, escalate to operator THEN (one decision, not a pre-built framework). Remove the symlink fallback entirely — if the atomic command is correct (Finding 1 fix applied), the window is zero; a symlink reintroduces the dual-path smell.
