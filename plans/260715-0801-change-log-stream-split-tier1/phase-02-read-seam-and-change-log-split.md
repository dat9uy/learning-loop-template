# Phase 2: Read seam and change-log split

## Overview

The core Tier 1 change. (1) Make the read chokepoint a **swappable projection seam** + extend it to load `meta-state.jsonl` and `change-log.jsonl` as one union (identity projection now; the projection function is the documented Tier-2 swap point). (2) Introduce `change-log.jsonl` as a true-append file with `merge=union`. (3) Route all change-log writes to it by `entry_kind`; non-change-log writes keep the existing table read-all→rewrite. (4) One-time migration of existing change-logs out of `meta-state.jsonl`. (5) Update raw-reading tests. Lands in a single PR with no parallel registry window.

## Requirements

- Functional: all registry reads see the union of both files; all `entry_kind=change-log` writes append to `change-log.jsonl`; all other writes behave unchanged on `meta-state.jsonl`.
- Non-functional: change-log writes are **true appends** (`appendFileSync` one line, then `invalidateCache`), not read-all→rewrite — both for `merge=union` safety and write cost. The projection is a pluggable function passed to `readRegistryWithCache` so Tier 2 swaps identity → last-wins-by-max-version without touching the chokepoint.
- Invariant: no code path mutates a change-log in place (immutability is what makes `merge=union` safe).

## Architecture

- **Read seam.** `readRegistryWithCache(root, parseFn)` already takes `parseFn` — that *is* the projection. Today `parseFn = _readAndParseRegistry` (loads `meta-state.jsonl`, `lines.map(parse)`). Change: `parseFn` loads **both** files, concats, returns entries (identity projection). The cache key expands from `meta-state.jsonl` mtime+size to **both files'** mtime+size (stat both; cache hit only if both match). `invalidateCache(root)` already deletes the root entry — ensure the change-log true-append writer calls it. **Red Team F5:** the current cache contract (`read-registry-cache.js:18`: "Caller must invoke invalidateCache(root) after any file write.") is FALSE post-split — the caller can invoke invalidate correctly and the cache still won't re-stat the second file. Phase 2 step 1 expands the cache contract doc + key composition.
- **Cold-tier cache SHA (Red Team F5).** `core/loop-introspect-cache.js:24-29` `registrySha256()` hashes ONLY `meta-state.jsonl`. Post-split this leaves the cold-tier cache stale on change-log appends. Phase 2 step 1 adds a `changeLogSha256(root)` helper, extends `readColdTierCache`/`writeColdTierCache` keys to include both SHAs (mirroring the paired atomic-read pattern at L51-69 used for `file-index.jsonl`). Acceptance: "cold-tier cache invalidates on `change-log.jsonl` append."
- **`consolidates` schema change (Validation Session 1 Q2).** Core schema (`core/meta-state.js:270`) currently declares `consolidates: z.string().optional()` with description "Comma-separated list of finding entry ids". The relationships tool (`meta-state-relationships-tool.js:21-25`) treats it as multi-valued. Phase 2 step 1 schema change: `consolidates: z.array(z.string()).optional()` and update the description. Phase 2 step 4 migration script converts any existing single-string `consolidates` value to a one-element array before partitioning into `change-log.jsonl` — preserves the relationships tool's documented behavior.
- **Projection swap point.** Extract the projection as a named function (e.g. `identityProjection(entries)`) with a code comment: "Tier 2 swaps this to `lastWinsByMaxVersion(entries)` (`group_by(.id) | max_by(.version)` semantics). The chokepoint stays untouched." Add a unit test that asserts the seam accepts an injected projection.
- **Write dispatch.** At the write layer (`writeEntry`/`appendRegistryEntryAtomic`), branch on `entry_kind`: `change-log` → `appendChangeLogEntryAtomic(root, entry)` (true append to `change-log.jsonl`); else → existing table write. **Red Team F2/F8(a):** the dispatch must run INSIDE `writeEntry`'s existing `enqueue(root, () => withRegistryLock(root, async () => ...))` wrapper (L761-802) — calling `appendChangeLogEntryAtomic` outside that wrapper loses cross-process lock coverage and two concurrent MCP servers can interleave byte-for-byte on the new file. Phase 2 also patches `metaStateBatch` (L1282-1293, L1297): the auto-emit change-log is currently pushed to the in-memory `entries` array and the whole file is rewritten via `persistRegistryAtomic`, NOT routed through `writeEntry`. Refactor `metaStateBatch` to call `appendChangeLogEntryAtomic` for the auto-emit (true-append to `change-log.jsonl`) so the migration invariant holds on day 1.
- **Immutability guard at core layer (Red Team F2/F7).** Plan previously scoped guards to handler-level (`meta-state-patch-tool.js:62-66`, `meta-state-resolve-tool.js:43-53`). That's insufficient — direct core callers (`tools/learning-loop-mastra/tools/handlers/scripts/fix-loop-design-refs.mjs:48`) bypass handlers. Add the `entry_kind !== "change-log"` guard at the FIRST LINE of core `updateEntry` (L812) and `archiveEntry` (L917) — handlers retain their existing guards. `merge=union` safety depends on this being enforced at the CORE layer.
- **Migration.** One-time script (or inline in the PR): **wrap the entire read-partition-rewrite window in `withRegistryLock(root, ...)` and call `invalidateCache(root)` after — Red Team F8(b).** Read `meta-state.jsonl`, **dedupe by id FIRST (Red Team F3): live file has 313 lines / 309 unique ids with 4 historical dup-id groups; partition by `entry_kind`, then within change-logs dedupe by id keeping `max_by(.version)` with documented tie-break (prefer later `created_at` for v=0 ties)**, write change-logs to `change-log.jsonl` (ordered), rewrite `meta-state.jsonl` without change-logs (ordered). Single PR on main, no concurrent registry PRs. Migration acceptance: "`change-log.jsonl` is free of intra-file duplicate ids."
- **Pre-migration advisory (Red Team F6).** Move the `.github/workflows/meta-state-pr-body-advisory.yml` path-filter + diff-command update INTO Phase 2 step 7. The migration PR is load-bearing and must be advised; if Phase 3 ships the update later, the migration PR sees only the `meta-state.jsonl` portion of the diff and the 220-entry change-log migration is invisible.
- **Ordering on the union (Red Team F15a).** Today the chokepoint returns entries in file-order (`meta-state.js:554-567`, no sort in `meta_state_list`). Post-split the union groups by file; the prior interleaved chronological view breaks. Phase 2 step 1 commits to: **post-concat sort by `created_at` ascending in `_readAndParseRegistry`** (or document the file-order-is-no-longer-chronological limitation in a code comment + the migration journal entry).
- **registry-table.sh multi-file default (Red Team F11a).** Phase 2 step 1 (or 1.5) extends `PATH_ARG` to accept multiple positional args and defaults to `meta-state.jsonl change-log.jsonl` after the split — keeps the operator's "zero relearning" promise.

## Risk Assessment

- **Highest risk: cache staleness across two files (Red Team F5).** If the cache key stays single-file, a `change-log.jsonl` append is invisible until `meta-state.jsonl` also changes → stale reads. Mitigation: dual-file stat in step 1; unit test that appends a change-log and asserts a fresh read sees it without touching `meta-state.jsonl`. **Cold-tier SHA at `loop-introspect-cache.js:24-29`** committed in step 1 (was deferred; now required for cold-tier cache invalidation).
- **Migration data loss / dup-id corruption (Red Team F3, F8b).** A botched partition could drop entries; the existing 4 dup-id groups in `meta-state.jsonl` would be preserved into `change-log.jsonl` and break the dedupe-by-id invariant. Mitigation: dedupe by id FIRST (keep `max_by(.version)`, prefer later `created_at` for v=0 ties); dry-run mode + count assertions + run on a clean checkout + wrap in `withRegistryLock` + `invalidateCache` after + commit the migrated files in the same PR so git history preserves the pre-migration state.
- **Test churn (Red Team F13).** **2 confirmed broken** (`meta-state-check-grounding-tool.test.js:243` parses line[1] as change-log; `meta-state-resolve-tool.test.js:46`); **3 secondary touches** (`cross-process-file-lock`, `drop-idempotency-cache`, `gate-recurrence`); **5 unaffected** (`connect-mcp-server-mutex`, `meta-state-log-change`, `meta-state-batch-tool`, `meta-state-consistency-check-tool`, `meta-state-relationships-dangling-refs`). Each broken test routes to chokepoint/`change-log.jsonl`; secondary tests verify still passing under split.
- **`merge=union` silently unsafe if a change-log is ever mutated (Red Team F2, F7).** Mitigation: **CORE-layer** immutability guard at `writeEntry` (L760-803), `updateEntry` (L812), `archiveEntry` (L917), `metaStateBatch` (write/update/delete/archive) — handler-level guards retained for resolve/patch paths. **8 guard sites, 8 tests.** Direct core callers (e.g. `fix-loop-design-refs.mjs:48`) bypass handlers, so core-layer is load-bearing for union safety.
- **`appendChangeLogEntryAtomic` lock coverage (Red Team F8a).** Must run INSIDE `writeEntry`'s existing `enqueue+withRegistryLock` wrapper; calling outside loses cross-process lock and two MCP servers can interleave byte-for-byte on `change-log.jsonl`. Mitigation: dispatch branch is inside the wrapper, not a sibling helper.
- **Ordering on the union (Red Team F15a).** Post-split the union groups by file; `meta_state_list` returns file-order which is no longer chronological. Mitigation: post-concat sort by `created_at` ascending in `_readAndParseRegistry`.
