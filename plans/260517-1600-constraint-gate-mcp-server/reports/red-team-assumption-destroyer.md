# Red Team Review: Constraint Gate MCP Server Plan

## Finding 1: Observation Schema Has No `constraint` Field — Gate Logic Cannot Match
- **Severity:** Critical
- **Location:** Phase 1 section "Requirements" (functional), Phase 3 section "Bash gating"
- **Flaw:** The plan's gate logic requires matching observations by `constraintType`. Phase 1 defines `checkObservationExists(constraintType, observations)` and Phase 3's bash-coordination-gate.cjs needs to "scan observation files for this constraint type." But `schemas/observation.schema.json` has NO `constraint` field. Its required fields are: `id`, `schema_version`, `type`, `status`, `created_at`, `updated_at`, `source_refs`. The `constraint` field exists only as non-schema-validated extra data in existing observation files (e.g., `observation-sandbox-cleanup-sudo-requirement.yaml` has `constraint: cleanup_requires_sudo`).
- **Failure scenario:** The MCP server's `check_gate` reads observation files and tries to match by constraint type. If it validates observations against the schema first (as Phase 2 requires for `record_observation`), the `constraint` field is invisible. If it reads raw YAML without schema validation, it works for existing files but new observations written via `record_observation` (Phase 2) would need the `constraint` field — which Phase 2 claims to validate against `observation.schema.json`. Either the schema is ignored (making Phase 2 validation meaningless) or the constraint field is lost during validation.
- **Evidence:** `schemas/observation.schema.json` defines only: `id`, `schema_version`, `type`, `status`, `created_at`, `updated_at`, `source_refs`, `notes`. No `constraint` field. Existing file `records/observations/observation-sandbox-cleanup-sudo-requirement.yaml:12` has `constraint: cleanup_requires_sudo` — not schema-validated.
- **Suggested fix:** Either extend `observation.schema.json` to include a `constraint` field (or a `constraints` array), or acknowledge that constraint matching operates on non-schema-validated YAML keys and document this as a design decision.

## Finding 2: `resource-budget.schema.json` Does NOT Extend `observation.schema.json`
- **Severity:** High
- **Location:** Phase 5 section "Requirements"
- **Flaw:** Phase 5 claims "Document inheritance: `resource-budget.schema.json` extends `observation.schema.json`" and adds a `$comment` saying "Extends observation.schema.json." This is factually false. The two schemas have zero shared required fields. `observation.schema.json` requires: `id`, `schema_version`, `type`, `status`, `created_at`, `updated_at`, `source_refs`. `resource-budget.schema.json` requires: `id`, `external_system`, `resource`, `budget`, `current`, `last_verified`, `verification_method`, `validation_window`. They share only `id`. There is no `$ref`, `allOf`, or any JSON Schema inheritance mechanism between them.
- **Failure scenario:** Phase 5 documents a false inheritance relationship. Future developers reading the docs will expect budget files to be valid observations and vice versa. Schema validation tools will not enforce this supposed relationship. Any code that assumes budget files can be treated as observation files (e.g., reading `type: observation` from a budget file) will fail because budget files don't have `type`, `schema_version`, `status`, etc.
- **Evidence:** `schemas/observation.schema.json` required fields vs `schemas/resource-budget.schema.json` required fields — only `id` overlaps. No `$ref` or `allOf` in either schema.
- **Suggested fix:** Remove the inheritance claim. Document that these are independent schemas that coexist in the `records/observations/` directory. If a shared base is desired, create a `base-record.schema.json` with the common `id` field and have both schemas extend it.

## Finding 3: CJS Hooks Cannot `require()` ESM `check-budget.js`
- **Severity:** Critical
- **Location:** Phase 3 section "Implementation Steps" step 3
- **Flaw:** Phase 3's bash-coordination-gate.cjs needs to "call check-budget.js (spawn child process)" for budget checks. `check-budget.js` is an ESM module (`import { readFileSync } from "node:fs"`) and the project's `package.json` has `"type": "module"`. The hook is CJS (`const fs = require('fs')`). The plan says "spawn child process" but Phase 1's `gate-logic.js` defines `checkBudgetState(system, resource)` as a function call — not a spawn. Phase 1 also creates `gate-utils.cjs` (CJS) that would need to call ESM code.
- **Failure scenario:** `gate-utils.cjs` (CJS) tries to `require()` or `import()` `check-budget.js` (ESM). In Node.js, CJS cannot synchronously `require()` ESM modules. Dynamic `import()` is async but hooks must be synchronous (the plan explicitly states "Hook must be synchronous — no MCP calls from hook"). The only option is `spawnSync`, but the plan doesn't specify this for `gate-utils.cjs`. Phase 1's `checkBudgetState` in `gate-logic.js` (ESM, since the MCP server is ESM) can call `check-budget.js` fine, but the CJS hook cannot reuse this logic.
- **Evidence:** `tools/check-budget/check-budget.js:1` uses `import` (ESM). `.claude/coordination/hooks/skill-coordination-gate.cjs:4` uses `require()` (CJS). `package.json` has `"type": "module"`. Phase 3 step 3 says "call check-budget.js (spawn child process)" but `gate-utils.cjs` (CJS) has no spawn mechanism specified.
- **Suggested fix:** Specify that `bash-coordination-gate.cjs` uses `spawnSync('node', ['tools/check-budget/check-budget.js', '--system', system, '--resource', resource])` for budget checks. Do NOT attempt to `require()` or `import()` ESM from CJS. Document this as a hard constraint: CJS hooks can only call ESM modules via child process spawn.

## Finding 4: No Profile Determination Mechanism for Edit/Write/Bash Hooks
- **Severity:** Critical
- **Location:** Phase 3 section "Requirements" and "Implementation Steps"
- **Flaw:** The coordination-config.json has three profiles (`code-generation`, `plan-execution`, `external-system`) with different `write_forbidlist` patterns. The existing `skill-coordination-gate.cjs` gets the profile from `registry.registered_skills[skillName].profile` — each registered skill has a hardcoded profile. But Phase 3's `write-coordination-gate.cjs` and `bash-coordination-gate.cjs` have no mechanism to determine which profile applies. The plan says "Read coordination-config.json → get active profile" but never specifies how the active profile is selected.
- **Failure scenario:** `write-coordination-gate.cjs` receives an `Edit` call with `file_path: "records/claims/foo.yaml"`. Should it use the `code-generation` profile (which forbids `records/**`) or the `plan-execution` profile (which allows `records/**`)? The hook has no context about which profile the agent is operating under. It would need to either: (a) always use the most restrictive profile (breaking legitimate plan-execution writes), (b) always use the least restrictive profile (defeating the purpose), or (c) have some external signal — which the plan doesn't define.
- **Evidence:** `skill-coordination-gate.cjs:48` reads `registered.profile` from the skill registry. `.claude/coordination/coordination-config.json` defines profiles but no mechanism to select one for non-Skill tools. Phase 3 never specifies profile selection logic.
- **Suggested fix:** Define a profile selection mechanism. Options: (a) default to most restrictive profile, (b) use an environment variable or context signal to select profile, (c) maintain a separate mapping of tool types to profiles, or (d) have the hook check if a Skill coordination is in progress (via a state file) and use that profile.

## Finding 5: Phase 2 Duplicate Detection Is Ambiguous and Overly Broad
- **Severity:** High
- **Location:** Phase 2 section "Requirements" and "Architecture"
- **Flaw:** Phase 2 says "Rejects duplicate observations (same constraint already recorded)" and the architecture shows "Check for existing observation with same constraint → return already_exists." But existing observation files contain MULTIPLE constraints in a single file. `observation-sandbox-cleanup-sudo-requirement.yaml` has three constraints: `cleanup_requires_sudo`, `stale_guard_deadlock`, `device_limit_blocks_reinstall`. The plan's duplicate detection by "constraint" is ambiguous: does it check by filename slug, by any constraint field in the YAML, or by the primary constraint?
- **Failure scenario:** Agent tries to record a new observation for `stale_guard_deadlock`. The system finds `observation-sandbox-cleanup-sudo-requirement.yaml` which already contains that constraint. Does it reject? The file's slug is `sandbox-cleanup-sudo-requirement`, not `stale-guard-deadlock`. If duplicate detection is by filename, it won't match. If by constraint field content, it will match but the file has 3 constraints — which one is "the constraint"? The plan doesn't specify.
- **Evidence:** `records/observations/observation-sandbox-cleanup-sudo-requirement.yaml` has `constraint: cleanup_requires_sudo`, `constraint: stale_guard_deadlock`, and `constraint: device_limit_blocks_reinstall` (lines 12, 17, 22). Phase 2 says "same constraint already recorded" without defining how to match.
- **Suggested fix:** Define duplicate detection precisely: (a) by filename slug (deterministic from constraint text), or (b) by scanning all observation files for a matching `constraint` field value. Option (a) is simpler and deterministic. Document that one observation file = one constraint, or specify that multi-constraint files are an existing pattern that the new system should handle.

## Finding 6: `check-budget.js` Exit Code Semantics Not Mapped to Gate Decisions
- **Severity:** High
- **Location:** Phase 1 section "Architecture" (step 4: `checkBudgetState`)
- **Flaw:** Phase 1 defines `checkBudgetState(system, resource)` → calls check-budget.js, and `makeGateDecision(...)` returns "escalate" when budget exhausted. But `check-budget.js` uses exit codes: 0 = ok, 1 = budget exhausted OR validation window active, 2 = error (no budget found, parse failure, schema invalid). The plan doesn't specify how these exit codes map to gate decisions. Exit code 2 could mean "no budget file exists" (should this be fail-open = ok?) or "budget file is malformed" (should this be fail-open = ok?).
- **Failure scenario:** `check-budget.js` returns exit code 2 because no budget file exists for the given system/resource. The gate logic interprets this as "budget exhausted" and returns "escalate." But the correct behavior (per the plan's "fail-open on missing files" constraint) should be "ok" — no budget constraint applies. Similarly, exit code 1 could mean either "budget exhausted" (escalate) or "validation window active" (different decision).
- **Evidence:** `tools/check-budget/check-budget.js:49` exits 2 for missing args, `:59` exits 2 for no budget files, `:69` exits 2 for parse failure, `:75` exits 2 for schema invalid, `:95` exits 1 for budget exhausted, `:98` exits 1 for validation window active. Phase 1 says "fail-open on missing/corrupt files" but doesn't map exit codes to decisions.
- **Suggested fix:** Define explicit mapping: exit 0 → "ok", exit 1 + stdout JSON `remaining <= 0` → "escalate", exit 1 + stdout JSON `validation_window.active` → "escalate" (or separate decision), exit 2 → "ok" (fail-open). Parse stdout JSON from check-budget.js to distinguish budget exhaustion from validation window. Document this mapping in Phase 1.

## Finding 7: Phase 4's `claude mcp add` Command Assumes CLI Availability
- **Severity:** Medium
- **Location:** Phase 4 section "Implementation Steps" step 1
- **Flaw:** Phase 4 says "Register MCP server via `claude mcp add --transport stdio --scope project constraint-gate -- node tools/constraint-gate/server.js`." This assumes the `claude` CLI is installed and available in PATH. The plan also provides a manual alternative (`.mcp.json`), but the primary instruction uses the CLI. If the CLI isn't available or the command syntax is wrong (the plan doesn't cite documentation for this command), the implementer will waste time debugging.
- **Failure scenario:** Implementer runs `claude mcp add` and gets "command not found" or an error about invalid syntax. They fall back to manual `.mcp.json` creation but don't know the correct format. The plan's manual example shows `.mcp.json` but the existing project doesn't have one — it uses `settings.json`. This creates confusion about which file to use.
- **Evidence:** No `.mcp.json` exists in the project. `.claude/settings.json` exists with hook config. Phase 4 provides both CLI and manual options but doesn't specify which is preferred or when to use which.
- **Suggested fix:** Make the manual `.mcp.json` approach the primary instruction (it's deterministic and doesn't depend on CLI availability). Keep the CLI command as an alternative. Specify that `.mcp.json` is project-scoped and shared with the team, while `settings.json` is local.

## Finding 8: `gate-utils.cjs` Duplication with MCP Server's `file-readers.js`
- **Severity:** Medium
- **Location:** Phase 1 section "Architecture" and Phase 3 section "Architecture"
- **Flaw:** Phase 1 creates `file-readers.js` (ESM) with `readCoordinationConfig()`, `readObservations(dir)`, `readSkillRegistry()`. Phase 3 creates `gate-utils.cjs` (CJS) with `readCoordinationConfig()`, `readObservations(dir)`, `matchConstraintPattern(command)`, `extractBashCommands(command)`, `globMatch(pattern, path)`. These are the same functions duplicated across CJS and ESM modules. The plan acknowledges this in Phase 3's risk table ("CJS can't import ESM gate-logic — Inline regex patterns in hook (small duplication)") but doesn't address the config/observation reading duplication.
- **Failure scenario:** A bug fix to `readCoordinationConfig()` in `file-readers.js` is not applied to `gate-utils.cjs` (or vice versa). The MCP server and hooks diverge in how they read config, leading to different gate decisions for the same input. This is the exact scenario the plan's "shared utilities via `require('./lib/gate-utils.cjs')`" was meant to prevent, but the sharing is only among CJS hooks — not between hooks and MCP server.
- **Evidence:** Phase 1 creates `file-readers.js` (ESM) with `readCoordinationConfig()`. Phase 3 creates `gate-utils.cjs` (CJS) with `readCoordinationConfig()`. Both read the same `coordination-config.json` file. Phase 3 risk table acknowledges "small duplication" but only mentions regex patterns, not config reading.
- **Suggested fix:** Accept the duplication as a hard constraint (CJS/ESM boundary), but document it explicitly. Add a comment in both files cross-referencing the other. Consider extracting the config reading logic into a standalone JSON file that both modules parse (no code duplication, just shared data format). Or create a shared CJS utility that both the hook and a thin CJS wrapper around the MCP server can use.

## Finding 9: Phase 3's `write-coordination-gate.cjs` Has No Profile Context for Forbidlist Matching
- **Severity:** High
- **Location:** Phase 3 section "Architecture" (write-coordination-gate.cjs steps)
- **Flaw:** Phase 3 says write-coordination-gate.cjs "Read coordination-config.json → get active profile" then "Check path against `write_forbidlist` patterns." But the coordination-config has THREE profiles with DIFFERENT forbidlists: `code-generation` forbids `records/**, evidence/**, docs/**, plans/**, schemas/**`; `plan-execution` forbids only `schemas/**`; `external-system` forbids only `schemas/**`. The hook has no way to know which profile is active. This is the same issue as Finding 4 but specific to the write gate's forbidlist matching.
- **Failure scenario:** Agent is executing a plan (plan-execution profile) and tries to `Edit` a file in `records/`. The hook uses the `code-generation` forbidlist (because it doesn't know the profile) and blocks the edit. The agent is stuck — it can't write to `records/` even though the plan-execution profile allows it. Alternatively, the hook uses the least restrictive profile and allows writes that should be forbidden.
- **Evidence:** `.claude/coordination/coordination-config.json` shows three profiles with different `write_forbidlist` arrays. Phase 3 never specifies how the write-coordination-gate selects a profile.
- **Suggested fix:** Same as Finding 4 — define a profile selection mechanism. For the write gate specifically, consider: (a) check if a `.bypass-next` or similar state file exists (like the skill gate uses), (b) use an environment variable set by the coordination system, or (c) default to `code-generation` (most restrictive) and require explicit bypass for other profiles.

## Finding 10: MCP Server `check_gate` Tool Interface Mismatches Hook's File-Based Interface
- **Severity:** Medium
- **Location:** Phase 1 section "Requirements" and Phase 3 section "Requirements"
- **Flaw:** Phase 1's `check_gate(action, target, context?)` takes `action` and `target` as separate parameters. Phase 3's bash-coordination-gate.cjs extracts `tool_input.command` as a single string. The MCP server's interface doesn't match the hook's interface for the same logical operation. When the agent calls `check_gate("bash", "docker run ubuntu")`, the MCP server needs to parse "docker run ubuntu" to extract the command. When the hook receives `Bash` with `tool_input.command: "docker run ubuntu"`, it has the raw command. The plan doesn't specify how `action` and `target` map to the hook's `tool_name` and `tool_input`.
- **Failure scenario:** Agent calls `check_gate("bash", "docker run")`. The MCP server's `matchConstraintPattern("bash", "docker run")` needs to match "docker" in the target. But the hook's `matchConstraintPattern("docker run")` matches against the full command string. If the MCP server's pattern matching is different from the hook's, they'll produce different gate decisions for the same logical operation — breaking the plan's goal of "same gate logic as an agent-callable tool."
- **Evidence:** Phase 1 defines `check_gate(action, target, context?)` with separate action/target. Phase 3's bash-coordination-gate.cjs extracts `tool_input.command` as a single string. Phase 1's `matchConstraintPattern(action, target)` takes two args; Phase 3's `matchConstraintPattern(command)` takes one.
- **Suggested fix:** Align the interfaces. Either: (a) `check_gate` takes a single `command` string (like the hook), or (b) define explicit mapping: `action` = tool name, `target` = tool_input field (command for Bash, file_path for Edit/Write). Document the mapping and ensure both MCP server and hooks use the same pattern matching logic.
