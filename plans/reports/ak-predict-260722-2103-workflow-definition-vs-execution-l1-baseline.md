# Prediction: define "workflow" (definition vs execution) at L1/L2 before auditing `run_workflow_*`

**Proposal:** Before auditing the ~10 `run_workflow_*` / `mastra_workflow_*` tools against the core-vs-Mastra transport rule, first define "workflow" at L1/L2 — splitting declarative *workflow definition* (ordering recipe, stateless, core-eligible data) from *workflow execution* (success-tracking, stateful by default, currently Mastra's job) — so the audit has a baseline, while keeping the core stateless (Functional Core, Imperative Shell).
**Trigger:** operator concern — "workflow" terminology is overloaded (definition vs execution); need a baseline to audit into; core must stay stateless.
**Key ground fact found during read:** `docs/loop-engine.md:57-63` (L1) already defines the relevant split as **deterministic-step vs agentic-step**, and explicitly states a workflow step can realize *either* role and "the concept role is primary; the mechanism is interchangeable." The baseline exists at L1 — under a different name.

## Verdict: CAUTION

A small L1/L2 *naming* clarification is warranted, but the bigger risk is **over-defining** (adding a 5th registry kind or a speculative workflow-definition schema) when the existing L1 vocabulary already suffices. Proceed: name, don't invent; audit per-workflow-class; let the audit result decide whether any *new* concept is needed.

## Agreements (all personas align)

- **The L1 baseline already exists.** deterministic-step (pure function, fixed outcome) vs agentic-step (calls model/operator, outcome recorded via promotion) IS the definition/execution distinction in disguise. "Workflow definition" = a declarative ordering of deterministic-steps (data); "workflow execution" = the imperative shell that tracks step success. No new first-class concept is required to *start* auditing.
- **"Workflow definition" must NOT become a 5th registry kind.** The 4-kind union (finding | change-log | rule | loop-design) is load-bearing; adding "workflow" to it is a schema change with wide blast radius. If workflow-definitions need to be durable+citable, they are **bound artifacts like skills** (operator-authored, change-log-gated, NOT agent-writable records), reusing the existing skill-migration pattern (L1 line 80). All personas reject a new record kind.
- **Execution home is per-workflow-class, not uniform.** Short deterministic sequences → core definition + agent-or-gate execution (stateless, transport-portable). Long-running agentic suspend/resume workflows → Mastra process state (stateful, MCP-only). Forcing all workflows into one home breaks either portability (pushing long ones to core) or correctness (pushing deterministic ones to a state machine they don't need).
- **Functional Core / Imperative Shell is already the actual architecture** — the runtime-state sidecar is file-based state, so "core stateless" really means "no *process* state; file-state is allowed." Workflow execution state may therefore live in the **runtime-state file** (still core-stateless in the process sense), not only in Mastra. This reframes the choice from 2 homes to 3: **agent | runtime-state file | Mastra process.**

## Recommendations

1. **Do NOT pre-build a workflow-definition schema or registry kind.** Add a one-paragraph L1 clarification in `docs/loop-engine.md` that names the existing split: *workflow definition = a declarative ordering of deterministic-steps plus per-step success contract (stateless data, bound-artifact-class like skills, NOT a record kind); workflow execution = the imperative shell that tracks live state against the definition, with one of three state homes — agent | runtime-state file | Mastra process.* This is the audit baseline. (L1 first because it is concept/vocabulary, L1's surface; L2 contract notes the transport consequence: agent/runtime-state homes are transport-portable, Mastra is MCP-only.)
2. **Audit the ~11 workflow tools per-workflow-class against the clarified L1 baseline**, tagging each: (a) deterministic short sequence → core definition + agent/gate execution (transport-portable, candidate to leave the Mastra residue); (b) long agentic suspend/resume → stays Mastra (MCP-only); (c) storage-bound (`workflow_storage_*`) → stays Mastra (`initStorage`). Let the audit's *result* decide whether any new L1 concept is needed — evidence-driven, not speculative.
3. **Treat the runtime-state file as the default durable-execution home**, not Mastra. The sidecar already carries ledger events / delivery attestations; step-success records are the same shape. This keeps "core stateless" true in the process sense while giving deterministic multi-step workflows durable, gate-enforced ordering without a state machine. Mastra is reserved for the narrow case that genuinely needs process-scoped suspend/resume.
4. **Keep workflow-definitions operator-authored bound artifacts** (skill-class, change-log-gated, not agent-writable) so an agent cannot rewrite step ordering to bypass a gate. Reuse the skill-migration ordering rationale (L1 line 80).
5. **Sequencing:** land (1) the L1 naming clarification + (2) the audit as a *report* before any re-homing code. The audit report is the evidence that justifies (or rejects) moving specific `run_workflow_*` tools out of the Mastra residue. No workflow tool changes code until the audit names which ones and why.

## Unresolved questions

- **Q1:** Does the runtime-state sidecar's schema need a `kind: "workflow_step"` (or similar) to carry per-step success records for deterministic multi-step workflows, or do existing `kind` values (ledger event / delivery attestation) already cover it? Answering this is part of the audit, not a prerequisite to it.
- **Q2:** For a deterministic sequence executed by the agent (home = agent), is "step N succeeded" *observed* by a gate reading runtime-state, or merely *asserted* by the agent's next tool call? The former is enforceable; the latter trusts the agent. This is the same abstraction-level question as the retracted Q-B (agent-behavior observation is runtime-state's job) — resolve it in the runtime-state layer's contract, not the transport layer.
- **Q3:** The 3 `mastra_workflow_*` helper tools (`generate_prompt`, `notify_artifact`, `trigger`) are a different surface from the 8 `run_workflow_*` — are they execution (Mastra-bound) or definition helpers (stateless, could be CLI)? The audit must classify them separately; the transport report's `MCP_RESIDUE` lumps them together today.
