# MCP-Client-Loading Slug Bloat ‚Äî Simplification Cascade

## Frame

After the four-part fix in commit `a9098dd` made the L2 probe deterministic, the visible artifact is 18 sibling `mcp-client-loading-missing-<HHMM>` entries in `meta-state.jsonl`. The user's question: "what is the solution then, there is a lot of redundant `*-mcp-client-loading-missing` that doesn't convey any meaning?"

Guiding principle (the cascade): if one insight eliminates multiple components, the components are all special cases of the same general pattern. The insight here is that **session-id-keyed findings are state machines, not event logs**. The slug misled the loop into treating them as events.

## Surface Area (what is actually in the registry)

`grep -c '"subtype": "mcp-client-loading"' meta-state.jsonl` returns 0 (the literal JSON key is `subtype: "mcp-client-loading"` ‚Äî confirmed via Grep on the 18 visible lines). The slug appears 18 times in the file (string match).

|-------|---------------------------|

**Operational meaning of the variation:** the L2 probe fired gap-open 18 times across 2 days. The runtime+layer+session_id is identical every time. The probes happened at different minutes. That is all.

## Cascade 1 ‚Äî The slug is decorative; the `(session_id, subtype, runtime, layer)` tuple IS the key

**Insight:** `tryClaimSessionId` in `tools/learning-loop-mcp/core/meta-state.js:531-578` already enforces uniqueness on `(sessionId, subtype, runtime, layer)`. The slug is generated by `entryBuilder()` *inside* the claim ‚Äî meaning the slug only gets used when the claim succeeds. The slug's timestamp comes from `new Date()` inside the same call that sets `created_at`. The slug adds ~25 bytes of "this happened at HH:MM" ‚Äî which `created_at` already carries in ISO-8601 form.

**Grep verification: no production code reads `mcp-client-loading-missing` as a key.** The 11 occurrences are split across:

- `tools/learning-loop-mcp/__tests__/gate-resolution-evidence.test.js` (4 call sites, lines 55, 88, 118, 460) ‚Äî uses it as a *cosmetic* slug in test fixtures.
- `tools/learning-loop-mcp/__tests__/cold-session-churn-regression.test.js` (5 call sites, lines 22, 46, 82, 100) ‚Äî same.
- `tools/learning-loop-mcp/__tests__/cold-session-discoverability.test.cjs` (2 call sites, lines 744, 940) ‚Äî the *one* place the slug is generated dynamically by the production probe.

The slug is read by `grep`. Nothing else.

**Collapse:** use a deterministic id for session-keyed findings. The key itself is descriptive enough.

```js
// before (per test fixture, 11 call sites):
id: core.generateId("mcp-client-loading-missing")

// after (deterministic, self-describing):
id: `meta-mcp-client-loading-${cli}-${layer}`
// e.g., "meta-mcp-client-loading-droid-L2"
```

The id is stable across runs. The session_id, subtype, runtime, and layer are still filterable as before ‚Äî the *consumers* of those filters don't read the id slug, they read the key fields.

**Net effect at the source:** the cold-session probe at line 940 generates the same id on every gap-open. `tryClaimSessionId` short-circuits to `claimed: false, existing: <existing entry>` because the key matches. The 18-entry log collapse to 1 entry per `(cli, layer)` pair.

## Cascade 2 ‚Äî `tryClaimSessionId` becomes a "reopen, not duplicate" state transition

**Insight:** with a deterministic id, gap-closed ‚Üí gap-open is a *state transition* on a single entity, not a new entity. The current dedup logic reads the registry, finds a match, returns the existing entry. With a deterministic id, the entry's `status` is the *only* varying field. The reopen semantic is: "stale entry with the same id + new gap-open evidence ‚Üí bump version, set status=`reported`, append gap-open timestamp to description."

This is the same pattern as `meta_state_re_verify` (SP0) for `expired ‚Üí reported` transitions, applied at the session-id layer. The loop already has the primitive; the cascade is "use it on session-id-keyed findings, not just on TTL-expired findings."

**Collapse (core/meta-state.js#tryClaimSessionId):**

```js
// after: deterministic id, state-transition dedup
export function tryClaimSessionId(root, key, entryBuilder) {
  return enqueue(root, () => {
    const entries = readRegistry(root);
    const match = entries.find((e) =>
      e.session_id === key.sessionId
      && e.subtype === key.subtype
      && (e.status === "active" || e.status === "reported")
      && e.description.includes(`runtime: ${key.runtime}`)
      && e.description.includes(`layer: ${key.layer}`),
    );
    if (match) return { claimed: false, existing: match };

    const entry = entryBuilder();
    const reopened = entries.find((e) => e.id === entry.id);
    if (reopened) {
      // Stale or resolved entry with same deterministic id ‚Üí reopen.
      // Bump version; refresh status, created_at, expires_at; append
      // gap-open evidence to description (audit trail).
      const updated = {
        ...reopened,
        status: "reported",
        created_at: entry.created_at,
        expires_at: entry.expires_at,
        resolved_at: null,
        resolved_by: null,
        description: `${reopened.description}\n[reopen ${entry.created_at}] ${entry.description}`,
        version: (reopened.version ?? 0) + 1,
      };
      // ... write back via existing append path with CAS on _expected_version
      return { claimed: true, id: entry.id, reopened: true };
    }
    // ... fresh write path unchanged
  });
}
```

**Net effect at the dedup layer:** the `_expected_version` CAS dance in test fixtures (cold-session-discoverability.test.cjs line ~913) becomes unnecessary ‚Äî same id, no version race. The 5 `_expected_version: existing.version ?? 0` call sites collapse to a simple `id`-keyed update.

## Cascade 3 ‚Äî The 18 existing entries collapse to 2 via one archival pass

**Insight:** a one-shot cleanup script groups the 18 entries by `(cli, layer)`, keeps the highest-version one as canonical, and supersedes the rest with a single `consolidated_into` change-log.

```js

// - supersede the other 16 with consolidated_into: <canonical id>

```

**Net effect at the registry layer:** 18 entries ‚Üí 2 (one per `droid` √ó `{L1, L2}` pair). The version chain on the canonical entry preserves the 18-event audit history (because Cascade 2's reopen logic appends gap-open evidence to `description`).

## Cascade 4 ‚Äî Meta-pattern: any test fixture with a session_id and stable key should have a stable id

**Pattern (3+ instances):**

- `mcp-client-loading-missing` (this report) ‚Äî 18 entries
- `stale-test` (cold-session-discoverability.test.cjs line 994) ‚Äî uses `generateId("stale-test")` with a stable `session_id`
- `target-finding` (gate-resolution-evidence.test.js line 459) ‚Äî uses `generateId("target-finding")` with a stable key

All three follow the same shape: a test fixture that creates a finding with a stable `session_id` and stable fields, then asserts behavior on that finding. The timestamp-based slug is purely cosmetic. The deterministic-id collapse applies to all three.

**Cascade rule for the project:** *if a fixture has a stable key (session_id, subtype, runtime, layer, or any other filterable field that uniquely identifies it across runs), give it a stable id.* Timestamp-based ids are for one-shot observations; key-based ids are for state machines.

## What does NOT change

- The `entry_kind: "finding"` schema ‚Äî fields are correct; only id-generation discipline is wrong.
- `tryClaimSessionId`'s read-and-check logic ‚Äî correct as dedup-by-key; the cascade is *upstream* (deterministic id) so the loop no longer needs to remember to call it to avoid duplicates.
- The visible-noise entries' information content ‚Äî they accurately describe 18 separate gap-open events. The bug is that the loop should not be logging 18 events for what is operationally 1 finding with 18 gap-open observations.
- The 4-part fix in `a9098dd` ‚Äî that fixed the probe flakiness (Layer 1, 2, 3, 4 of the change-log). This cascade is Layer 5: the *representation* of the gap-open events the probe logs.

## Why this is the answer, not just an answer

The cascade eliminates 3 things with one insight:
1. The 18-entry JSONL bloat (visible noise).
2. The 5 `_expected_version` CAS dances in test fixtures (latent complexity).
3. The semantic mismatch between "session-keyed finding" and "timestamped event" (root confusion).

The insight is small ("session-id-keyed findings are state machines"). The cascade is the standard "everything is a special case of one general pattern" simplification. The previous fix (deterministic slug, hardcoded runtime+layer in description, soft-delete on gap-close) was already 70% of the way there ‚Äî it just kept the timestamp slug, which is the one part that doesn't pay for itself.

## One-liner summary

Stop using `generateId("mcp-client-loading-missing")` in session-keyed test fixtures. The slug duplicates the `(session_id, subtype, runtime, layer)` key. Use a deterministic id like `meta-mcp-client-loading-${cli}-${layer}` and let `tryClaimSessionId` do reopen-on-version-bump, not duplicate-write. The 18 entries collapse to 2 (one per cli√ólayer pair) with full history preserved on the surviving entry's version chain.

## Red flags avoided

- "Quick fix: just delete the 18 stale entries" ‚Äî would have lost the audit trail and re-introduced the churn on next run.
- "Add a one-off filter to `loop_describe` cold tier" ‚Äî treats the symptom (visible noise) not the cause (timestamp slug).
- "Increase the dedup window" ‚Äî already at the structural limit; widening it doesn't help when the dedup key is the timestamp slug itself.
- "Archive them via `meta_state_archive`" ‚Äî already possible, but the cascade is the *preventive* fix; archival is a one-time cleanup, not a structural answer.
