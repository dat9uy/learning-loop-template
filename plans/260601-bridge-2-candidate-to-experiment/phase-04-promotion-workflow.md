# Phase 4: Promotion Workflow

## Overview

Define the minimal human-driven promotion workflow for moving candidate assertions through the status chain: `candidate` → `pending_approval` → `active`. This phase is intentionally minimal — full automation (class-level approval, auto-promotion on experiment success) is Bridge 3/4 territory.

## Requirements

- Functional: Document the promotion workflow so operators know how to move a candidate assertion forward
- Functional: Ensure validation layer allows `pending_approval` references (already true — Layer 5 only blocks `candidate`)
- Functional: Ensure `extract-index` maps `passed` evidence to `active` (already true — `STATUS_MAP.passed → active`)
- Non-functional: No autonomous promotion — every transition requires explicit human approval

## Architecture

### Promotion Workflow Diagram

```
┌─────────────┐     Human review      ┌─────────────────┐     Experiment runs      ┌─────────┐
│  candidate  │ ────────────────────▶ │  pending_approval  │ ────────────────────▶ │  active  │
│             │    (operator decides  │                    │    (evidence passes    │         │
│  (extracted │     worth testing)    │  (experiment draft  │     → extract-index   │         │
│   from doc) │                        │   approved, ready) │      updates status)   │         │
└─────────────┘                        └─────────────────┘                        └─────────┘
```

### Transition 1: candidate → pending_approval

**Trigger:** Human operator reviews the candidate assertion and the experiment draft generated by the mapping tool.
**Decision:** Is this assertion worth testing? Does the experiment plan look correct?
**Action:** The human operator edits the assertion YAML file directly:
```yaml
# In records/<surface>/index/<assertion-id>.yaml
status: pending_approval
```
**Validation:** `validateRecords` allows `pending_approval` references (Layer 5 only blocks `candidate`).
**Note:** In a future version, this could be an MCP tool (`record_update_assertion_status`) with operator_confirmation. For v1, manual edit is sufficient.

### Transition 2: pending_approval → active

**Trigger:** The experiment runs successfully and the evidence file is recorded with `validation_status: passed`.
**Action:** Run `pnpm extract:index` (or `index_extract` MCP tool). The extractor reads the evidence file, sees `validation_status: passed`, and updates the assertion status:
```js
// In index-entry-builder.js
const STATUS_MAP = {
  passed: "active",
  pending: "candidate",   // Bridge 1
  failed: null,
  draft: null,
};
```
**Result:** The assertion is now `active` and can be referenced by product code, decisions, and experiments without triggering the candidate block.

### Transition 2b: pending_approval → rejected (failure path)

**Trigger:** The experiment fails or contradicts the assertion.
**Action:** Evidence is recorded with `validation_status: failed`. `extract-index` skips the entry (returns null, no YAML written). The operator may manually archive the assertion.
**Note:** A future version could add a `rejected` status to the enum. For v1, removal + `failed` evidence is sufficient.

### What This Phase Does NOT Build

- **No auto-promotion on experiment success** — deferred to Bridge 3/4
- **No class-level approval pattern** — deferred to Bridge 3
- **No MCP tool for status updates** — manual edit for v1; tool can be added later

## Implementation Steps

1. Document the promotion workflow in `docs/artifact-concepts.md`:
   - Add a "Candidate Promotion Workflow" section
   - Describe the two transitions (candidate → pending_approval, pending_approval → active)
   - Describe the failure path (pending_approval → rejected via failed evidence)
   - Emphasize that both transitions require human action
2. Verify `validateRecords` allows `pending_approval` references:
   - Run `pnpm validate:records` — should pass
   - Confirm no existing records violate this
3. Verify `extract-index` maps `passed → active`:
   - Run `pnpm extract:index --dry-run` — should show existing entries unchanged
   - Confirm `STATUS_MAP` in `index-entry-builder.js` is correct
4. Create a test fixture with a `pending_approval` assertion referenced by an experiment:
   - Add to `candidate-block.test.js` or create a new test
   - Verify `validateRecords` does NOT flag `pending_approval` references

## Risk Assessment

- **Manual edit is error-prone:** Medium — operators might edit the wrong field. Mitigation: clear documentation and a future MCP tool.
- **No `rejected` status in enum:** Low — `failed` evidence + skipped extraction achieves the same effect. A `rejected` status could be added later.
- **Promotion workflow is too minimal:** Expected — this is the design. Full automation is deferred.
