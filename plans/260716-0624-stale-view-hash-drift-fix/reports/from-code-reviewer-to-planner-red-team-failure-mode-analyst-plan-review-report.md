## Finding 1: `meta_state_re_verify` upsert placed BEFORE `applyUpdateAndCheck` writes an orphan file-index baseline on CAS conflict — data-loss class
- **Severity:** Critical
- **Location:** Phase 03 Architecture block (lines 68-100), specifically the comment "Inside handler, after the verification loop, before applyUpdateAndCheck"
- **Flaw:** The plan's diff inserts the `upsertFileIndexEntry(root, canonical, currentHash)` call BEFORE `applyUpdateAndCheck` returns `ok: true`. On a CAS conflict (operator A's `_expected_version` mismatches the current version because operator B re-verified concurrently), `applyUpdateAndCheck` returns `{ok: false, reason: "version_mismatch", current_version}` (see `core/update-entry-helpers.js:29-32`). The handler then early-returns with `re_verified: false`. BUT the file-index has already been mutated: `currentHash` (bytes A read) is now stored. Operator B's earlier `upsertFileIndexEntry` already stored `currentHash_B` for the same canonical key — both writes succeed (no CAS on the index; `meta-state.js:787-794` performs NO version check). Net result: the index baseline reflects a hash that no operator actually committed via a verified re-verify, and the entry's `last_verified_at` was NOT stamped. On the next sweep, `hasDrifted` reads `currentHash === storedHash` (both reading the same fresh bytes), so drift branch is silent — the entry looks grounded against a hash that was never operator-attested.
- **Failure scenario:** Two operators (or one operator + one CI re-verify) race. Operator A: `meta_state_re_verify({id, _expected_version: 3})`. Operator B has just done the same with `_expected_version: 3` and won. A's verification steps pass; A's `computeFileHash` reads bytes X; A's `upsertFileIndexEntry` writes X to the index. A's `applyUpdateAndCheck` returns `version_mismatch` (current_version: 4). A returns `re_verified: false`. The index now has X for that key. The entry's actual hash may be Y (if the file was edited between A's read and B's write). Next sweep reads Y vs X → drift fires. Or worse: next sweep reads X vs X → drift silent → operator thinks entry is grounded, but no verification actually completed for A's invocation.
- **Evidence:** `tools/learning-loop-mastra/core/meta-state.js:787-826` (`upsertFileIndexEntry` performs key-shape + hash validation only, NO version check). `tools/learning-loop-mastra/core/update-entry-helpers.js:29-32` (CAS failure returns `{ok:false, reason:"version_mismatch"}` — handler MUST not have side-effected index). Plan line 84 places the upsert before the CAS guard.
- **Suggested fix:** Move the upsert AFTER `applyUpdateAndCheck` returns `{ok:true, entry}`; skip on `{ok:false}` regardless of reason. The plan's "best-effort" framing at line 119 is incorrect — best-effort means "we tried but failed silently," not "we wrote before checking if the entry was actually updated."

## Finding 2: PR body table format silently fails CI advisory's 6-section regex — every required section emits a warning
- **Severity:** Critical
- **Location:** Phase 04 lines 42-55 ("The PR body must satisfy `rule-pr-body-registry-deltas`")
- **Flaw:** The plan's PR body uses one section (`## Registry deltas (per rule-pr-body-registry-deltas)`) with a single Markdown table. The actual CI advisory `.github/workflows/meta-state-pr-body-advisory.yml:71-78` checks for six separate top-level headings: `## Swept entries`, `## Resolved entries`, `## New entries`, `## Promoted rules`, `## Superseded entries`, `## Archived entries` — each matched by regex `^## <name>\s*\n([^#]+)`. The plan's table-format will fail all 6 regex checks; the workflow emits `core.warning("PR body section \"Swept entries\" is empty or missing")` × 6.
- **Failure scenario:** Plan lands. Author writes PR body per plan (one table, no six headings). CI advisory runs and emits 6 warnings to the PR Checks tab. Reviewer (or operator) sees 6 warnings and assumes the registry deltas were missed. They may block the PR or ask for a body rewrite, blocking ship. The plan claims "format mismatches will surface there" (line 107) — true, but the surface is 6 misaligned warnings, not "the table format was rejected."
- **Evidence:** `.github/workflows/meta-state-pr-body-advisory.yml:71-78` matches exactly 6 headings via `new RegExp("^## Swept entries\\s*\\n([^#]+)", 'm')` etc. Plan lines 47-55 produce `## Registry deltas (per rule-pr-body-registry-deltas)` followed by a table — none of the 6 expected headings match.
- **Suggested fix:** Phase 04 must enumerate six `## X entries` headings with non-empty bodies (even when content is "(none) — see comment"), or omit sections explicitly (the regex requires the heading to be followed by content; "## Swept entries\n" alone with no body fails). Add a CI dry-run step that runs `tools/scripts/ci-registry-deltas.sh /tmp/registry.diff` locally + greps the PR body for the 6 headings before sending.

## Finding 3: Cold-tier test reads `fileIndex` at module-load — `pnpm test:iter` and `pnpm test:debug` skip the seed step, so post-Phase-3 the cap can blow on a stale index without any actual drift
- **Severity:** Critical
- **Location:** Plan.md Verification protocol (lines 138-149); Phase 03 line 99-105 (cold-tier-regression.test.js diff); Phase 03 Risk Assessment (lines 134-138)
- **Flaw:** Phase 3 removes the `fileIndex: new Map()` workaround and adds `codeHashes: computeCurrentHashes(current.all_findings, root)`. The test file loads `fileIndex` at module-load (`cold-tier-regression.test.js:19`: `const fileIndex = readFileIndex(root);`) — before any test runs. The seed step (`seed-file-index.mjs`) re-hashes every mc:true path and is invoked by `pnpm test` (`package.json:17`), `pnpm test:cold-session` (line 20), and `gate:self-verify` (line 24). It is NOT invoked by `pnpm test:iter` (line 18: `vitest run --bail=1` only), `pnpm test:debug` (line 19: targeted debug tests), or `pnpm check:freshness` (line 21: freshness suite). After Phase 3, if any of those scripts is run on a dirty worktree (e.g., a developer ran Edit on a file, or `gate:self-verify` rewrote the index but the developer also did manual `meta_state_refresh_file_index` calls), `computeCurrentHashes` reads CURRENT bytes (the dirty bytes) and compares against the STALE index baseline → drift fires for every mc:true finding whose file was edited → `derivedStaleMc.length` jumps to ~80+ → cap (currently 16 in the workaround, 2 post-fix placeholder) blows → CI red.
- **Failure scenario:** Developer runs `pnpm test:iter` locally to triage a flake. Phase 3 has landed. Their worktree has 3 files edited since last `pnpm test`. `computeCurrentHashes` reads the 3 edited files → 3 drift signals. Pre-fix, this was masked by `fileIndex: new Map()`. Post-fix, the cap test fires `assert.ok(derivedStaleMc.length <= <N+2>)` and fails with "Phase 7: derived-stale cap broken". Developer now thinks the cap regression was reintroduced and starts a wasteful investigation. The plan's mitigation (line 135: "measure on a clean tree with pnpm test first; the seed step normalizes the index. If the cap is still flaky, expand the headroom to 3 instead of 2") does not address the actual root cause: `test:iter` is a documented escape hatch that doesn't seed, and Phase 3 changes the cap's preconditions.
- **Evidence:** `package.json:17-21` shows seed runs ONLY on `pnpm test` and `pnpm test:cold-session`. `tools/learning-loop-mastra/__tests__/legacy-mcp/cold-tier-regression.test.js:19` is module-level `readFileIndex(root)`. Plan's Phase 3 removes the empty-Map workaround that was the explicit mitigation for the no-seed case (cold-tier-regression.test.js:86-98 comment block).
- **Suggested fix:** Either (a) extend `package.json:18-21` so `test:iter`, `test:debug`, `check:freshness` prepend the seed step; OR (b) make the cold-tier test itself tolerant of stale index by computing `codeHashes` from the seeded baseline (mirror the seed step's behavior at module-load); OR (c) defer the cold-tier test from `pnpm test:iter` (move it to a separate `pnpm test:cold` script that runs the seed step first). Option (a) is the smallest blast radius.

## Finding 4: `meta_state_re_verify` index refresh has no `gate-log.jsonl` entry on success — drifts the audit trail contract that `meta_state_refresh_file_index` (the canonical path) upholds
- **Severity:** High
- **Location:** Phase 03 lines 78-100 (re_verify refresh block); Plan.md constraint on line 119 ("if the file is missing or unreadable at refresh time, skip the refresh and log a `gate-decision.log` line")
- **Flaw:** The audited path `meta_state_refresh_file_index` writes a `gate-log.jsonl` entry on EVERY successful refresh — `meta-state-refresh-file-index-tool.js:114-122` calls `appendGateLog(root, { timestamp, tool: "meta_state_refresh_file_index", path, code_fingerprint, refreshed_at, findings_regrounded, reason })`. The plan's `meta_state_re_verify` path only logs on the failure branch (its `catch` block writes `action: "index_refresh_skipped"`). Successful refreshes are silent. After Phase 3, an operator who runs `meta_state_re_verify` to clear drift has no audit trail of (a) who triggered the re_verify, (b) which file's hash was rewritten, (c) why. The seed step's `seed-file-index.mjs` is also silent (`loop-get-instruction-tool.js:64`: "seed-file-index.mjs is a mechanical bulk re-seed that intentionally omits per-path gate-log entries (git history is its audit)"). The plan's re_verify path falls into the silent bucket — but unlike seed (which is operator-initiated bulk), re_verify is per-finding and is the DOCUMENTED user-facing API for clearing drift. The audit gap is a real data-integrity regression: operators can't answer "who cleared the drift signal on entry X at time Y?"
- **Failure scenario:** Operator re-verifies 3 findings via the agent. Drift clears. Two weeks later, an audit asks "when did the file-index.jsonl entry for `tools/learning-loop-mastra/core/stale-view.js` last get rewritten?" The `git log` shows no commit (file-index.jsonl is gitignored). The `gate-log.jsonl` shows only `meta_state_refresh_file_index` entries from the operator's earlier manual refreshes — none from the agent's re_verify calls. Audit fails. The plan's failure-branch log only fires when the file is missing — the success branch, which is the common path, leaves no breadcrumb.
- **Evidence:** `tools/learning-loop-mastra/tools/handlers/meta-state-refresh-file-index-tool.js:114-122` (audited path). `tools/lib/gate-logging.js:74-91` (the append helper is cheap, no I/O blocking on failure). Plan's Phase 03 success branch (lines 80-96) has no `appendGateLog` call.
- **Suggested fix:** In the success branch of Phase 03's re_verify refresh, call `appendGateLog(root, { timestamp, tool: "meta_state_re_verify", action: "index_refreshed", id, canonical, code_fingerprint: currentHash, findings_regrounded: 1 })`. The cost is one JSONL line per re_verify pass; the gain is a complete audit trail. Mirrors `meta_state_refresh_file_index`.

## Finding 5: `computeCurrentHashes` cache-invalidation race: re_verify's `upsertFileIndexEntry` invalidates the `readFileIndex` cache, but `computeCurrentHashes` was called BEFORE the upsert — so the same handler invocation produces an inconsistent view
- **Severity:** High
- **Location:** Phase 03 lines 78-96 (re_verify block); Phase 01 lines 67-82 (`computeCurrentHashes` design)
- **Flaw:** In `meta_state_re_verify`, the handler flow becomes: (1) `readFileIndex(root)` → Map (cached at module-load of `core/meta-state.js:716`). (2) Compute `codeHashes` via `computeCurrentHashes` (or skip — only `applyUpdateAndCheck` runs after). (3) Verification loop runs. (4) `upsertFileIndexEntry(root, canonical, currentHash)` succeeds → `_invalidateFileIndexCache(root)` (line 822). (5) Return. NEXT call to `meta_state_sweep` or `meta_state_query_drift` in the same process: `readFileIndex(root)` re-reads the file (cache invalidated). The re-verify's own handler chain doesn't re-read the cache — so within one invocation, no inconsistency. BUT: if the plan extends `computeCurrentHashes` to other consumers (Phase 02 list: sweep, relationship-validate, relationships), those handlers call `readFileIndex(root)` AND `computeCurrentHashes(entries, root)` at handler start (lines 33-38 of Phase 02's `meta-state-sweep-tool.js` diff). If a `meta_state_re_verify` runs CONCURRENTLY in the same process (MCP server is multi-tool-capable, parallel tool calls are possible via `meta_state_batch` or multi-agent), the upsert from re_verify invalidates the cache; the consumer's `readFileIndex` and `computeCurrentHashes` may interleave. Specifically: consumer reads fileIndex (cached, stale); re_verify upserts (cache invalidated); consumer reads codeHashes (fresh bytes); consumer compares currentHash ≠ storedHash (which still has old hash because cache was invalidated AFTER read) → drift fires for an entry that was just re-verified. This is a read-cache race introduced by the cache invalidation in `upsertFileIndexEntry` (line 822).
- **Failure scenario:** Two concurrent MCP calls: `meta_state_re_verify({id: meta-X})` and `meta_state_sweep({})`. Both share the same process and the same `_fileIndexCache`. Sweep reads fileIndex at T0 (gets cached Map). Re_verify upserts at T1 (invalidates cache). Sweep calls `computeCurrentHashes` at T2 (reads fresh bytes). Sweep's `isStaleView` compares: storedHash (from cached Map, OLD) vs currentHash (fresh, NEW). Drift fires for meta-X. Operator sees drift signal on a finding they just re-verified. Operator re-re-verifies. Drift clears. Loop repeats on every concurrent call.
- **Evidence:** `tools/learning-loop-mastra/core/meta-state.js:716-823` shows the `_fileIndexCache` Map and the `_invalidateFileIndexCache` invalidation in `upsertFileIndexEntry`. Plan's Phase 02 wires 3 consumers to call `readFileIndex(root)` then `computeCurrentHashes(entries, root)` at handler start (Phase 02 lines 33-58). Plan's Phase 03 adds `upsertFileIndexEntry` to re_verify, which triggers cache invalidation.
- **Suggested fix:** Either (a) make the cache invalidation lazy — re-validate the cache at READ time, not at write time, by checking the file's mtime/size on every read (already done at line 750-754, but the check uses `_fileIndexCache.get(root)` BEFORE the new upsert's invalidation takes effect within the same microtask); OR (b) document that consumers must call `readFileIndex(root)` AFTER their `computeCurrentHashes` call, not before (counterintuitive but eliminates the race). Option (a) is the right fix — make `readFileIndex` re-stat the file every call and bypass cache when stat diverges.

## Finding 6: `meta_state_resolve` consult-gate `rule-no-orphaned-evidence` reads `fileIndex` AT RESOLVE TIME — re_verify clears the gate, but a subsequent file edit between re_verify and resolve re-arms the gate. Plan's Phase 4 doesn't address this

The plan's Phase 4 step `meta_state_check_grounding({id: "meta-260716T0603Z-…"})` (line 106) verifies the entry's GROUNDING is `grounded` AT THE CHECK MOMENT — but does NOT verify it stays grounded through the resolve call.

## Finding 7: Seed step's `readFileIndex(root)` at line 68 may return a stale cached Map from a prior concurrent process — cross-process cache leak masks the "completeness" check at line 82-85
- **Severity:** Medium
- **Location:** Plan.md claim 5 (line 119-120, "the seed step normalizes the index"); Phase 03 Risk Assessment line 136-137 ("upsertFileIndexEntry race with seed-file-index")
- **Flaw:** `seed-file-index.mjs` runs `readFileIndex(root)` at line 68 to compute `finalIndex.size` and assert completeness (line 82-85: `if (finalIndex.size < existingKeys.length) { console.error('INCOMPLETE'); process.exit(1) }`). If a parallel `pnpm test` invocation is mid-run (vitest workers are child processes; `pnpm test:cold-session` is one such), each process has its own `_fileIndexCache` Map (module-scope). Process A (vitest cold-tier test) loads fileIndex at module spawn (T0). Process B (a parallel `meta_state_refresh_file_index` operator action via MCP) upserts (T1), invalidating B's cache only. Process A's cache still has the T0 Map; A's module-load happens to read the file at T0 fresh (not cached across processes). So no cross-process leak in vitest — vitest workers are spawned with a fresh module-load state. BUT: in a long-running MCP server process (server.js), the fileIndex cache persists across tool calls. If the server is invoked to refresh_file_index (T1) and then someone runs `node tools/scripts/seed-file-index.mjs` in a SEPARATE shell (T2), seed's readFileIndex (T2) reads the disk fresh — no cross-process leak in that direction. Where the leak happens: the long-running MCP server's cached Map is NOT invalidated by seed-file-index.mjs's writes (because seed writes via `upsertFileIndexEntry`, which DOES invalidate via `_invalidateFileIndexCache(root)` at line 822). So cross-process is OK. WITHIN process: server's cache is invalidated by seed's upserts because both go through `upsertFileIndexEntry`. So no leak in normal use. BUT: in the plan's Phase 3 cold-tier-regression.test.js (line 19: `const fileIndex = readFileIndex(root);` at module-load), the test thread's cached Map is captured BEFORE seed runs (because vitest spawns workers, then the test file's module-load fires). If seed is invoked at T1 AFTER the test thread loads, the test thread's cached Map is stale; the test uses the stale Map for `derivedStaleSet(entries, {fileIndex})`. `computeCurrentHashes` reads fresh bytes; comparison fails; drift fires. This is the SAME race as Finding 3 but with a different mechanism — module-load order rather than missing seed step.
- **Failure scenario:** CI runs `pnpm test`. Seed-file-index completes at T0. Vitest spawns 4 worker threads. Thread 1 (cold-tier-regression.test.js) module-loads at T1 (after T0 — cache miss, reads fresh fileIndex from disk). Thread 1 runs its test at T2. Within T1-T2, another parallel worker (Thread 2, say meta-state-sweep test) calls `meta_state_re_verify` (with `META_STATE_VERIFY_EXEC=1`), which upserts file-index.jsonl at T1.5. Thread 1's `readFileIndex(root)` was called at T1 (before the upsert), so its cached Map is STALE. Thread 1's `computeCurrentHashes` at T2 reads fresh bytes; the cached Map has the pre-re_verify baseline; drift fires for the re_verified entry. Test fails. Same race applies to other test files that do `const fileIndex = readFileIndex(root)` at module-load.
- **Evidence:** `tools/learning-loop-mastra/__tests__/legacy-mcp/cold-tier-regression.test.js:19` is module-scope. `tools/learning-loop-mastra/core/meta-state.js:716` (`_fileIndexCache` is process-local). Phase 3 wires `meta_state_re_verify` to upsert (line 84), invalidating the cache.
- **Suggested fix:** (a) Change the cold-tier test to re-read fileIndex INSIDE the test body, not at module-load. (b) Add a `readFileIndexFresh(root)` helper that bypasses the cache. (c) Move the seed step to run AFTER vitest worker spawns (not in `package.json:17` but inside a vitest globalSetup). Option (c) is the cleanest fix.

## Finding 9: Rollback hole — if Phase 03 ships but the cap precompute is wrong, the cold-tier test fails with no documented rollback path. Plan offers "expand headroom to 3" but that's a workaround, not a rollback
- **Severity:** Medium
- **Location:** Phase 03 Risk Assessment line 135 ("If the cap is still flaky, expand the headroom to 3 instead of 2")
- **Flaw:** The plan's mitigation for cap-regression is to widen the headroom — but the cold-tier-regression test was added to catch a specific class of regression (per the comment at cold-tier-regression.test.js:86-98: "Why age-only: the pretest seed re-hashes every mc:true cited path to its CURRENT bytes... the path-keyed hasDrifted would flag every grounded finding as stale... capping that count trips on every newly-filed finding"). Widening the headroom doesn't restore the forcing function — it just delays the alarm. The original age-only cap at 16 (precompute 14 + 2 headroom) was a deliberate forcing function: each newly-filed finding that drifts adds 1 to the count; cap trips at 16 newly-drifted. Post-fix, the cap at 2 (or 3) is "drift unmeasurable post-seed" — by definition, drift count should be 0 every run. So the cap is not a forcing function anymore; it's a tripwire for the unknown (real drift). Plan provides no rollback path: if Phase 3 reveals that hash-aware drift produces wildly different counts than expected (e.g., 50+ drift entries), the operator has no documented recovery — they can either (a) bump the cap to 50+ and silently accept degraded signal, or (b) revert Phase 3 and re-introduce the age-only workaround. Plan does not mention option (b).
- **Failure scenario:** Plan ships. CI runs. `derivedStaleMc.length` reports 7 (some mc=true findings drift, some age-stale, both contribute). Cap is 2. CI red. Operator looks at plan, sees "expand headroom to 3" → sets cap to 7 (precompute + 2). CI green. But the cap is now meaningless: 7 < 7 passes even with 7 real drift events. ## Finding 11: The `computeCurrentHashes` function silently swallows ALL errors (EACCES, EMFILE, EISDIR, ENOSPC, etc.) — `meta_state_sweep` returns `stale_view_count` that does not distinguish "no drift" from "could not read N files"
- **Severity:** Medium
- **Location:** Phase 01 lines 78-79 (`try { out.set(canonical, computeFileHash(absPath)); } catch { /* missing → no signal */ }`); Plan.md claim 1 (line 88)
- **Flaw:** Plan's `computeCurrentHashes` helper has a bare `catch { /* missing → no signal */ }`. The comment says "missing → no signal" but the catch swallows ALL errors, not just `FileNotFoundError`. `computeFileHash` (check-grounding.js:81-88) only throws `FileNotFoundError` for `!existsSync`. If the file exists but `readFileSync` throws (EACCES, EMFILE, EISDIR on a directory of the same name, ENOSPC at the syscall level, EBUSY on Windows), the error propagates as a generic Error. Phase 01's helper catches it silently. Result: `codeHashes` doesn't have an entry for that path; `hasDrifted` returns `false` (no signal); `derivedStaleSet` excludes the entry from drift; sweep's `stale_view_count` is artificially low. Operator runs sweep, sees 0 drift, concludes all grounded — but 5 files were unreadable. The unreadable files are EXACTLY the ones most likely to need re-verify (permissions changed, mounted FS issue, deleted mid-run).
- **Failure scenario:** Operator runs `meta_state_sweep` after a filesystem event (chmod on a directory, NFS mount hang, deletion of an intermediate file). 5 of 80 mc=true cited paths return EACCES. `computeCurrentHashes` silently skips them → `stale_view_count = 0`. Operator assumes all grounded. `meta_state_check_grounding({id})` for one of the affected entries: `code_ref_exists: false, status: "drifted", drift_kind: "code_missing"`. Discrepancy: sweep said no drift, check_grounding said drifted. Operator's mental model breaks. No gate-log entry from sweep to explain the discrepancy.
- **Evidence:** `tools/learning-loop-mastra/core/check-grounding.js:81-88` shows `computeFileHash` throws raw Error for non-ENOENT cases. Plan's `computeCurrentHashes` (plan.md lines 67-83) uses bare `catch { /* missing → no signal */ }` — both wording and behavior are wrong for non-ENOENT errors.
- **Suggested fix:** Distinguish error classes. In `computeCurrentHashes`: `if (err instanceof FileNotFoundError) skip; else { appendGateLog(root, { tool: "computeCurrentHashes", canonical, error_code: err.code, error_message: err.message }); skip }`. Don't throw (keeps handler robust) but emit a signal so the operator can diagnose silent skew. Optionally return a sidecar Map of `{canonical → error}` so the sweep tool can surface `unreadable_count` in the report.

## Finding 12: `computeCurrentHashes` reads `entries` (the entire registry) but the cold-tier test passes `current.all_findings` — verifying the post-fix drift-stale count requires sweeping EVERY entry, but the pre-fix workaround only ran on `mc:true` findings (the filter at line 102)
- **Severity:** Medium
- **Location:** Plan.md Phase 03 Architecture lines 51-55 (cold-tier diff); cold-tier-regression.test.js line 102 (`(f) => f.mechanism_check === true || f.mechanism_check === null`)
- **Flaw:** Phase 3's cold-tier diff calls `derivedStaleSet(current.all_findings, { ... codeHashes: computeCurrentHashes(current.all_findings, root) })`. The post-filter at line 102 narrows to mc=true or mc=null. So `codeHashes` is built for ALL findings' cited paths, but only mc=true/null findings are scored. Phase 3 reads ~80+ files (every distinct cited path across all findings), but the score only matters for ~80 mc=true/null entries. Wasted I/O: ~268 findings × ~80 unique paths = up to 268 readFileSync calls (the plan's dedup brings this to ~80 distinct paths, but `computeCurrentHashes` is invoked with all entries). For 80 distinct paths, each readFileSync is ~ms-scale on SSD; total ~100ms. Acceptable. BUT: `computeCurrentHashes` does NOT skip resolved/superseded findings — it reads files for findings that won't be scored. The plan's "filter at the post-set" wastes reads.
- **Failure scenario:** (a) Performance: `meta_state_sweep` handler now does N readFileSync per invocation. For 268 findings × 80 paths = ~80 reads (after dedup). Per the plan's Risk Assessment in Phase 02 (line 127): "for 268 findings with ~80 distinct paths, this is ~80 reads per sweep. Acceptable for an MCP tool (not in hot path)." The plan acknowledges this — but the cold-tier test runs on every `pnpm test`, and adding 80 reads to the test critical path matters when CI is flaky. (b) Wasted I/O: filtering at the post-set means `computeCurrentHashes` reads files for resolved/superseded findings whose drift is irrelevant. The plan should filter `entries` to mc=true/null BEFORE computing `codeHashes`.
- **Evidence:** Plan.md Phase 03 lines 51-55 (the diff), Phase 02 lines 127-128 (the "acceptable" claim), cold-tier-regression.test.js line 102 (the post-filter).
- **Suggested fix:** Pre-filter `entries` to mc=true/null before passing to `derivedStaleSet` (and before building `codeHashes`). The helper signature should accept a pre-filtered array. Saves reads for resolved/superseded findings (whose drift is irrelevant — they're terminal).
