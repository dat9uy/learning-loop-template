# Phase 4: recurrence-tracker

## Overview

The fourth and final piece of the bash-gate-debate plan. The recurrence tracker reads the cross-surface `.gate-decision.log` (from Phase 3), groups entries by `rule_id + command_prefix_normalized`, and auto-files `meta_state_report` findings when a group exceeds the threshold (N≥3 occurrences in M≤10 minutes).

**Why a tool, not embedded in the bash gate:** the gate shouldn't call `meta_state_report` directly. That's a layering violation (same as direct file I/O on `meta-state.jsonl` — see finding `meta-260606T2102Z-...`). The tool is the canonical path; the agent/operator calls it explicitly, and the SessionStart hook can call it as part of the discoverability surface.

**Why this is the loop's self-model:** the recurrence tracker is the meta-surface action that closes the learning loop. The gate observes → the decision log records → the recurrence tracker detects patterns → the meta-state registry captures them → the operator/agent refines the rule or accepts the bypass. This is the "debatable + self-improving" meta-pattern from the planning-order report.

## Requirements

Functional:
- New file: `tools/learning-loop-mcp/core/recurrence-tracker.js` — exports `findRecurrentGroups(root, options)` and `checkAndEmit(root, options)`.
- New MCP tool: `tools/learning-loop-mcp/tools/gate-check-recurrence-tool.js` with `gate_check_recurrence` name.
- New test: `tools/learning-loop-mcp/__tests__/gate-recurrence.test.js` (4 tests).
- Grouping key: `rule_id + command_prefix_normalized`, where:
  - `command_prefix` is the first 50 chars of the command.
  - `_normalized` removes single + double quotes and collapses whitespace.
  - Example: `node -e "console.log('foo')"` and `node -e "console.log('bar')"` normalize to the same key.
- Threshold: N≥3 occurrences in M≤10 minutes for the same group. These defaults are constants in `core/recurrence-tracker.js`; can be tuned per call.
- Dedup: if a `meta_state_report` finding already exists with the same `command_prefix_normalized` (matched by `evidence_code_ref` = `tools/learning-loop-mcp/core/recurrence-tracker.js` + a custom field like `recurrence_key`), the tracker does NOT emit a duplicate.
- Auto-file: emits a finding with:
  ```json
  {
    "id": "meta-<auto-generated>",
    "entry_kind": "finding",
    "category": "gate-logic-bug",
    "severity": "warning",
    "affected_system": "gate-logic",
    "subtype": "recurring-false-positive",
    "description": "Pattern recurred N times in M minutes: <rule_id> + <command_prefix_normalized>. First seen: <first_ts>. Last seen: <last_ts>. Sample commands: [...]",
    "evidence_code_ref": "tools/learning-loop-mcp/core/recurrence-tracker.js",
    "mechanism_check": true,
    "status": "reported",
    "expires_at": "<now + 24h>"
  }
  ```
- Manifest updates: add to `tools/manifest.json` and `agent-manifest.json`.
- The new SessionStart hook `tools/learning-loop-mcp/hooks/recurrence-check-on-start.js` calls `checkAndEmit(root)` once per session start. This is the canonical hook integration (decided in Validation Session 1); `inbound-gate.js` is a UserPromptSubmit hook and is not the right binding. The hook is a thin wrapper around the `gate_check_recurrence` MCP tool; the tool is the canonical path (the agent can also call it explicitly).

Non-functional:
- The tracker is read-only against the decision log; it does not mutate it.
- The tracker writes to `meta-state.jsonl` ONLY via the `meta_state_report` MCP tool's logic (or a direct `appendFileSync` if MCP tools are not available; the same pattern as the existing meta-state code). The tracker does NOT call `meta_state_report` directly from the gate (the gate is not the actor; the SessionStart hook is).
- The threshold is configurable via a constant in the file: `RECURRENCE_THRESHOLD_N = 3`, `RECURRENCE_WINDOW_MS = 10 * 60 * 1000`. Documented in JSDoc.
- The tracker skips the auto-file step in test environments (env var `GATE_RECURSION_DRY_RUN=1` makes the tool return the groups without writing).

## Architecture

### Reader (in `core/recurrence-tracker.js`)

```js
// tools/learning-loop-mcp/core/recurrence-tracker.js
import { readFromAllSurfaces, SURFACES } from "./surfaces.js";
import { readDecisionLog } from "./gate-decision-log.js";
import { readRegistry } from "./meta-state.js";

const RECURRENCE_THRESHOLD_N = 3;
const RECURRENCE_WINDOW_MS = 10 * 60 * 1000;

/**
 * Normalize a command prefix for grouping.
 * First 50 chars; remove single + double quotes; collapse whitespace.
 */
function normalizePrefix(command) {
  return command
    .slice(0, 50)
    .replace(/['"]/g, "")
    .replace(/\s+/g, " ")
    .trim();
}

/**
 * Find all (rule_id, command_prefix_normalized) groups that exceed the threshold
 * within the configured window.
 */
export function findRecurrentGroups(root, options = {}) {
  const threshold = options.threshold ?? RECURRENCE_THRESHOLD_N;
  const windowMs = options.windowMs ?? RECURRENCE_WINDOW_MS;
  const sinceTs = Date.now() - windowMs;

  // Read entries from all surfaces
  const allEntries = readDecisionLog(root, { since: new Date(sinceTs).toISOString() });
  
  // Group by (rule_id, command_prefix_normalized)
  const groups = new Map();
  for (const entry of allEntries) {
    if (!entry.rule_id) continue; // skip block-without-rule entries
    const key = `${entry.rule_id}::${normalizePrefix(entry.command_prefix)}`;
    if (!groups.has(key)) groups.set(key, []);
    groups.get(key).push(entry);
  }

  // Filter to groups exceeding the threshold
  const recurrent = [];
  for (const [key, entries] of groups) {
    if (entries.length >= threshold) {
      const [rule_id, prefix] = key.split("::");
      recurrent.push({
        rule_id,
        command_prefix_normalized: prefix,
        count: entries.length,
        first_ts: entries[0].ts,
        last_ts: entries[entries.length - 1].ts,
        sample_commands: entries.slice(0, 3).map((e) => e.command_prefix),
      });
    }
  }
  return recurrent;
}

/**
 * Check for recurrent groups and auto-file findings (deduped against existing).
 * Returns { checked_groups: N, findings_emitted: M, recurrent: [...] }.
 */
export function checkAndEmit(root, options = {}) {
  const recurrent = findRecurrentGroups(root, options);
  
  // Dedup: skip groups that already have an active/reported finding
  const existing = readRegistry(root).filter(
    (e) => e.entry_kind === "finding"
      && e.subtype === "recurring-false-positive"
      && (e.status === "active" || e.status === "reported")
      && e.recurrence_key,
  );
  const existingKeys = new Set(existing.map((e) => e.recurrence_key));
  const fresh = recurrent.filter((g) => !existingKeys.has(`${g.rule_id}::${g.command_prefix_normalized}`));

  // Emit findings (unless dry-run)
  const dryRun = process.env.GATE_RECURSION_DRY_RUN === "1";
  if (!dryRun) {
    for (const group of fresh) {
      appendFileSync(join(root, "meta-state.jsonl"), JSON.stringify({
        id: `meta-${new Date().toISOString().replace(/[^0-9]/g, "").slice(0, 15)}Z-${slugify(group.command_prefix_normalized)}`,
        entry_kind: "finding",
        category: "gate-logic-bug",
        severity: "warning",
        affected_system: "gate-logic",
        subtype: "recurring-false-positive",
        recurrence_key: `${group.rule_id}::${group.command_prefix_normalized}`,
        description: `Pattern recurred ${group.count} times in ${Math.round((new Date(group.last_ts) - new Date(group.first_ts)) / 60000)}min: ${group.rule_id} + "${group.command_prefix_normalized}". First: ${group.first_ts}. Last: ${group.last_ts}. Samples: ${group.sample_commands.join(" | ")}`,
        evidence_code_ref: "tools/learning-loop-mcp/core/recurrence-tracker.js",
        mechanism_check: true,
        status: "reported",
        created_at: new Date().toISOString(),
        expires_at: new Date(Date.now() + 24 * 60 * 60 * 1000).toISOString(),
      }) + "\n", "utf8");
    }
  }

  return { checked_groups: recurrent.length, findings_emitted: dryRun ? 0 : fresh.length, recurrent };
}
```

The `slugify` helper is imported from `core/slugify.js` (existing utility). The `appendFileSync` to `meta-state.jsonl` is the same pattern used elsewhere in the codebase; the tool's actor is the SessionStart hook, not the bash gate.

### MCP tool

```js
// tools/learning-loop-mcp/tools/gate-check-recurrence-tool.js
import { z } from "zod";
import { checkAndEmit } from "#mcp/core/recurrence-tracker.js";
import { resolveRoot } from "#lib/resolve-root.js";

export const gateCheckRecurrenceTool = {
  name: "gate_check_recurrence",
  description: "Check the gate's decision log for recurring false-positive patterns and auto-file findings...",
  schema: {
    threshold: z.number().int().positive().optional().describe("Minimum occurrences to emit (default 3)"),
    window_minutes: z.number().int().positive().optional().describe("Time window in minutes (default 10)"),
  },
  handler: async ({ threshold, window_minutes }) => {
    const root = resolveRoot();
    const result = checkAndEmit(root, {
      threshold,
      windowMs: window_minutes ? window_minutes * 60 * 1000 : undefined,
    });
    return { content: [{ type: "text", text: JSON.stringify(result) }] };
  },
};
```

### SessionStart hook integration (decided in Validation Session 1)

Add a new SessionStart hook — NOT wire into `inbound-gate.js` (which is a UserPromptSubmit hook, not SessionStart). The integration:

1. **New universal hook:** `tools/learning-loop-mcp/hooks/recurrence-check-on-start.js` (~30 lines). Reads stdin (Claude Code / Droid CLI SessionStart payload), calls `checkAndEmit(root)`, logs the result to stderr (so the operator sees "recurrence-tracker: emitted N finding(s)" on session start). Exits 0 unconditionally (the hook is informational, not gating).
2. **Thin CJS wrappers** (parallel to the existing `inbound-state-gate.cjs` pattern):
   - `.claude/coordination/hooks/recurrence-check-on-start.cjs` — `execFileSync` the universal hook.
   - `.factory/coordination/hooks/recurrence-check-on-start.cjs` — same.
3. **Surface settings** — add the SessionStart event binding to `.claude/settings.json` and `.factory/hooks/settings.json` (or equivalent). The wrapper exists; the binding wires it.

This makes the recurrence tracker run on every session start. The hook is a thin wrapper around the tool; the tool is the canonical path (the agent can also call it explicitly via `gate_check_recurrence`).

## Risk Assessment

| Risk | Mitigation |
|---|---|
| Auto-filed findings flood `meta-state.jsonl` | The threshold is 3 occurrences in 10min; the dedup-by-`recurrence_key` prevents duplicates. The 24h TTL ensures stale findings are swept. |
| `command_prefix_normalized` collapses too aggressively (false positive grouping) | The first 50 chars + quote-removal is conservative. Two different commands with the same first 50 chars (and no quotes) would group together — acceptable, the operator can review the sample commands. |
| `command_prefix_normalized` doesn't collapse enough (false negative) | The first 50 chars is short enough that quoted variations get normalized away. The recurrence tracker is a heuristic, not a precise matcher. |
| Concurrent auto-file emissions create duplicate findings (race between read-time dedup and write) | **Validation Session 1 decision:** accept duplicates. Threshold is N≥3; concurrent > 3 calls hitting the same rule in the same 10min window is rare. Operator de-dupes via `meta_state_resolve` if it happens. Simpler implementation. |
| `checkAndEmit` writes to `meta-state.jsonl` from the SessionStart hook (not the agent/operator actor) | Per the report (Position 1C analysis), the loop's self-model is itself an actor for the meta-surface. The SessionStart hook is the loop's self-initiated action; the agent/operator can also call the tool explicitly. The `evidence_code_ref` pin keeps the finding's lineage clear. |
| Threshold tuning requires a code change | Acceptable for Phase 4. A future discoverability work (Step 4 Phases 4-5) could expose the threshold as a runtime parameter. |
| `findRecurrentGroups` is O(N log N) for N entries in the window | The window is 10 minutes; a heavy session produces ~100-1000 entries in the window. The sort is fast. No perf concern at this scale. |
| `recurrence_key` is a new field on the finding entry; not in the canonical schema | The meta-state schema (per `core/meta-state.js`) is open; new fields are accepted. The field is documented in the finding's `description` and the recurrence tracker's JSDoc. |
| `recurrence-check-on-start.js` fires on every Claude Code / Droid CLI session start; if both surfaces are running, it fires twice | The hook is idempotent (read-then-check-then-write). Even if it fires twice, the second call sees the first's emission and is a no-op. The rare-duplicate risk (above) is the worst case. |

## Security Considerations

- The tracker reads from `coordination/.gate-decision.log` (operator-mediated) and writes to `meta-state.jsonl` (the meta-surface). Both are existing, audited paths.
- The tracker does NOT bypass the gate; it observes and reports. The gate remains the authoritative decision surface.
- The auto-filed findings are `severity: "warning"` (not `escalate`); they don't trigger the gate's escalation path. The operator/agent decides whether to act.
- The tracker's `command_prefix_normalized` is a heuristic, not a precise matcher. The sample commands in the finding's `description` give the operator enough context to verify.
