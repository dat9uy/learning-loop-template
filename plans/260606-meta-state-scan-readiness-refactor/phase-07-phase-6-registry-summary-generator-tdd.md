# Phase 7: Refactor #7 — `docs/registry-summary.md` Generator + Warm-Tier `registry_summary` Field

## Overview

Extends `meta_state_sweep` to emit `docs/registry-summary.md` (auto-generated, never edited) on every run. Adds a `registry_summary` field to `loop_describe` warm tier (computed inline, no file I/O). The summary is a relationship-and-coverage surface: counts, coverage stats, top-referenced entries, recent drift.

## Requirements

- **Functional**: `meta_state_sweep({ apply: false, emit_summary: true })` returns the markdown body; committing the file to `docs/registry-summary.md` works; `loop_describe({ tier: 'warm' })` returns a `registry_summary` field with the same content inline.
- **Non-functional**: the summary has 4 sections (Counts, Coverage, Top references, Drift); the `last_generated_at` field updates on re-run; the summary is idempotent on the registry (re-running sweep produces the same content modulo the timestamp); the warm tier field is O(N) compute (no file I/O).

## Architecture

### Sweep extension

`tools/meta-state-sweep-tool.js` gains an optional `emit_summary: boolean` parameter. When `true`, after the existing sweep logic, the handler:
1. Calls `readRegistry(root)`.
2. Calls `buildInverseIndexes(entries)` (Phase 4) to get the inverse maps.
3. Calls `summarize` (Phase 6) for each entry.
4. Renders the markdown via a pure function `renderRegistrySummary({ entries, indexes, now })`.
5. Writes the markdown to `docs/registry-summary.md` (path configurable, default).
6. Emits a `change-log` entry documenting the sweep (only if `apply: true`).

```
meta_state_sweep({ apply: true, emit_summary: true })
  → existing sweep logic (expiry + auto-resolve)
  → if emit_summary:
    → entries = readRegistry(root)
    → indexes = buildInverseIndexes(entries)
    → markdown = renderRegistrySummary({ entries, indexes, now: new Date() })
    → writeFileSync(docs/registry-summary.md, markdown)
  → return { swept: N, summary_emitted: true, summary_path: "docs/registry-summary.md" }
```

### Warm-tier field

`tools/loop-describe-tool.js` warm tier gains:
```js
registry_summary: renderRegistrySummary({ entries: warmEntries, indexes: warmIndexes, now: new Date() })
```

`renderRegistrySummary` is exported from a shared module (e.g., `core/registry-summary.js`, new) so both the sweep and the warm tier call the same function.

**Format of `docs/registry-summary.md`:**

## Risk Assessment

- **Risk**: the sweep writes to a file outside the meta-state registry, potentially bypassing the gate. → **Mitigation**: the sweep tool is a planned write (operator-approval required); the file path is configurable (default `docs/registry-summary.md`); the path is restricted to under `docs/` to avoid arbitrary writes.
- **Risk**: the markdown format is hardcoded and hard to extend. → **Mitigation**: the renderer is a pure function in its own module; sections can be added/removed by editing the renderer. The format is documented in the function's docstring.
- **Risk**: a human edits `docs/registry-summary.md` and the next sweep overwrites their edits. → **Mitigation**: the file's first line is `_Auto-generated by meta_state_sweep on YYYY-MM-DD. Do not edit._` — explicit warning. If human edits are a recurring need, future plan can move to a `.template.md` extension.
- **Risk**: the warm tier `registry_summary` field is large (~1-2KB) and increases warm tier token cost. → **Mitigation**: warm tier is already small (~5K tokens); +1-2KB is acceptable. The field is opt-in via tier (warm is the only tier that includes it). If warm tier cost becomes a concern, the field can be made opt-in via a `with_summary: true` flag (out of scope for this plan).
- **Risk**: the sweep extension slows down `meta_state_sweep` for operators who don't want the summary. → **Mitigation**: the `emit_summary` flag defaults to `false`; existing callers see no diff. The cost is opt-in.
