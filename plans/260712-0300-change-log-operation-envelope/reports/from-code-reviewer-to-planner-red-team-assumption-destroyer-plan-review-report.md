## Finding 1: `meta_state_log_change` MCP schema silently strips `operation_envelope` — RED tests never go RED through the tool layer
- **Severity:** Critical
- **Location:** Phase 1, Step 3 (RED test (a)) and Step 4 (RED test (b)); Plan frontmatter "`metaStateChangeEntrySchema` accepts an optional top-level `operation_envelope` field" acceptance criterion.
- **Flaw:** The plan assumes extending the underlying `metaStateChangeEntrySchema` (meta-state.js:148-189) is sufficient for `callTool("mastra_meta_state_log_change", {...operation_envelope...})` to round-trip the field. It does not. `meta-state-log-change-tool.js:31` builds the MCP input shape via `metaStateChangeEntrySchema.pick(MIGRATED_FIELDS).shape`, and `MIGRATED_FIELDS` (lines 16-26) does **not** include `operation_envelope`. Zod's default for `z.object({...}).pick({...})` is `.strip()` on unknown fields (the schema is not declared `.strict()`). So the MCP wire path drops the field silently. The registry entry will never carry the envelope regardless of Phase 1 GREEN.
- **Failure scenario:** Phase 1 step 3 RED test (a) — even before any code change — the `callTool` succeeds and returns `{logged:true, id}`; the registry entry has no `operation_envelope` field; the test fails on its assertion, not on callTool throwing. Phase 1's "GREEN" merely makes the underlying schema permissive; the MCP tool still strips. Test (b) likewise fails on assertion, not on the SyntaxError the plan predicts.
- **Evidence:** `tools/learning-loop-mastra/tools/handlers/meta-state-log-change-tool.js:16-31`; schema `metaStateChangeEntrySchema` at `core/meta-state.js:148-189` is not `.strict()`.
- **Suggested fix:** Add `operation_envelope: true` to `MIGRATED_FIELDS` in `meta-state-log-change-tool.js:16-26`. Treat this as a Phase 1 prerequisite, not a separate concern. Also explicitly add `.strict()` to `metaStateChangeEntrySchema` if the RED-via-throw semantics are required.

## Finding 3: "Idempotency" hash does not detect duplicate execution; naming is semantically false
- **Severity:** Critical
- **Location:** Plan § Architecture row "Idempotency"; Phase 1 step 6 (RED test (d)); Schema field at Phase 1 Architecture `idempotency: z.string().regex(/^sha256:[a-f0-9]{64}$/)`; Acceptance criteria row "Idempotency hash is stable across re-runs of the same ops on the same registry snapshot".
- **Flaw:** The plan excludes `version` from the hash input (correct to make hash stable) — but the same ops applied to the same target entries produce the same hash whether or not they have already run. Re-running a write-batch twice (different newly-minted entry ids each time) produces two registry states with identical `idempotency` hashes. The hash is a property of the op-list + entry-id-set, not of whether the work has already landed. It is not idempotency; it is a content-hash. Using this for "tests can assert idempotency across retries" (Plan § Idempotency row) misrepresents what the assertion proves: a retry of the same-content batch on the same-target always matches, even when the registry diverged.
- **Failure scenario:** Test framework re-runs the post-migration test twice (e.g. via `pnpm test --watch`). Run 1: pre-existing entry + new writes, hash H1. Run 2: same target entry, but registry now contains duplicates from run 1, hash still H1. Assertion `idempotency === H1` passes — the registry is now corrupted. The fix masks double-applies.
- **Evidence:** Plan § "Idempotency hash construction" Phase 1 step 4 of architecture: "Canonicalize the registry snapshots: extract entry-id-sets, sort lexicographically." No mention of distinguishing "applied once" vs "applied twice"; canonicalization would produce identical sets only if no writes have already created entries.
- **Suggested fix:** Either (a) rename to `content_hash` (literal description of the field) and drop the "idempotency" framing throughout the plan; or (b) include `created_at` of the auto-emitted log in the hash input, so each emission has a distinct hash AND include the previously-emitted envelope's hash as a "previous_hash" linkage to detect chained re-application.

## Finding 4: Auto-emitted change-log ID is unspecified — silently relies on `generateId` availability and slug uniqueness
- **Severity:** High
- **Location:** Phase 2, Architecture § "Batch handler" step 1: `{id: <new meta-id>, ..., change_target: envelope.target, ...}`; Implementation Step 9.
- **Flaw:** The plan's Phase 2 auto-emit code uses the placeholder `<new meta-id>` with no concrete id-construction strategy. The only available helper is `generateId(slug)` at `core/meta-state.js:935`. If the implementer picks `slugify(envelope.target)`, two batches with the same `target` (e.g., multiple "drift-closeout" runs) collide, and `meta_state_batch.update` then overwrites the previous envelope instead of creating two distinct change-log entries. If the implementer picks a per-batch unique suffix (e.g., timestamp + random), tests that assert "the most recent change-log with `operation_envelope.kind === 'migration'`" still work — but idempotency re-runs (Finding 3) double-write entries.
- **Failure scenario:** Two `meta_state_batch` envelopes with `target: "test-migration"` in the same test file → both auto-emit change-logs with the same `id` → registry `writeFileSync` fails validation OR (worse) overwrites and only one change-log persists. The first emit's `pre_count` / `post_count` data is lost. Test (e) green today, but Phase 2 integration tests with two consecutive batches with the same target will collide.
- **Evidence:** `core/meta-state.js:935` `generateId(slug)` produces deterministic ids from slug; `metaStateBatch` write loop at `core/meta-state.js:760-836` rejects no duplicate-id writes at the schema layer (entry.id is `optional()` in `metaStateChangeEntrySchema` at line 149) — so duplicate ids silently overwrite.
- **Suggested fix:** Define the auto-emit id-construction rule in the plan (e.g., `meta-{ISO timestamp}-{random suffix}` or compose target+batch-write-hash). Add a duplicate-id check in `metaStateBatch` auto-emit step before append.

## Finding 5: `pre_count.by_status` keys are not aligned with the current schema enum
- **Severity:** High
- **Location:** Plan § Architecture "Change-log schema" row; Phase 1 Architecture Zod snippet; loop-design description quoted in `meta-state.jsonl:277`.
- **Flaw:** The loop-design description enumerates `by_status:{open,active,stale,resolved,superseded}` — but `metaStateFindingEntrySchema.status` enum (`core/meta-state.js:106-107`) is the collapsed `{open, resolved, superseded}` per plan 260707-0812. Legacy registry entries may still carry `active|reported|stale` (per the lifecycle migration in progress). The Phase 1 Zod snippet declares `by_status: z.record(z.string(), z.number())` (open dict) — Zod doesn't constrain keys. So `buildEnvelope` will produce whatever categories the registry actually contains, but the test (a) assertion will check specific keys like `{open: 22, resolved: 0, ...}` and may fail because the live registry has `active` or `stale` keys.
- **Failure scenario:** Pre-migration registries (live today) carry `active`/`reported`/`stale` statuses. `buildEnvelope` returns `by_status: {active: 22, stale: 12, ...}`. Test asserts `by_status.open === 22` (the spec reflects the post-migration lens) — fails. Or worse, the test author accepts the registry's actual keys and the field becomes a registry-shape-dependent snapshot rather than a stable contract.
- **Evidence:** `core/meta-state.js:106-107` (status enum collapsed) and `:870-878` (backward-compat code mapping `active|reported|stale` to `open` on read); `meta-state.jsonl:271` (the finding) describes pre-migration state as "10 active + 12 stale".
- **Suggested fix:** Specify the exact by_status keys (canonical set, e.g., `{open, resolved, superseded}`) AND the by_kind keys (canonical set `{finding, change-log, rule, loop-design}`) in the Zod snippet using `z.enum(...)` not `z.record(...)`. Add a normalization pass at `buildEnvelope` boundary (call `matchesStatusFilter` semantics) so legacy entries fall into the canonical bucket.

## Finding 6: `BATCH_SIZE_LIMIT` discrepancy between handler (500) and core (100 default) is unaddressed

- **Location:** Phase 2, Architecture § "Batch caller" + Implementation Step 8 (modify `meta-state-batch-tool.js`).
- **Flaw:** `meta-state-batch-tool.js:7` defaults `BATCH_SIZE_LIMIT = 500`; `core/meta-state.js:741` defaults `BATCH_SIZE_LIMIT = 100` (overridable via env). `META_STATE_BATCH_LIMIT` env applies to BOTH (the env is read at module-load time in both files). Phase 2 Step 8's "extend schema" does not reconcile them. The auto-emit logic the plan adds in Phase 2 cannot fire because the batch was rejected before the ops loop — but the `envelope` field is valid. Result: handler returns a misleading `applied:0` without explaining why; tests must special-case.
- **Evidence:** `tools/learning-loop-mastra/tools/handlers/meta-state-batch-tool.js:7, 40`; `core/meta-state.js:741, 751`.
- **Suggested fix:** Define the canonical `BATCH_SIZE_LIMIT` in a single shared module (e.g., `core/constants.js`) and import on both sides. Or document the env-var dependency in the plan and require both files to honor the same default.

## Finding 7: "Auto-emit uses the same `enqueue` + `withRegistryLock` pattern" — claim is vacuous; actual atomicity depends on inner-loop behavior
- **Severity:** High
- **Location:** Phase 2, Architecture § "Batch handler" step 1; Non-functional requirement "Auto-emit uses the same `enqueue` + `withRegistryLock` pattern as the batch itself (no race window)".
- **Flaw:** The plan's auto-emit does NOT acquire a new lock — it appends to the same in-memory `entries` array inside the already-held `withRegistryLock`. The "same pattern" claim refers to the outer lock, not a fresh acquisition. The plan's Risk row ("Auto-emit appends to the same in-memory entries array before the file rewrite, atomically") is correct that the append is part of the same lock cycle, but it does NOT handle the writeFileSync-throws-after-append case: if the rebuild of `tmpPath` fails (disk full, EACCES), the rollback branch (lines 824-829) reads `preBatchContent` from the OLD file, then `writeFileSync(path, preBatchContent, "utf8")` — but that path overwrite happens AFTER `entries` has been mutated to include the auto-emit. The "rollback" actually leaves an in-memory-and-on-disk divergence until... wait, no, the rollback DOES revert the file. So far so good. But: the `renameSync(tmpPath, path)` at line 840 happens after the ops loop completes. If THAT throws (e.g., intermediate path issue), the auto-emit is silently lost. The plan does not address this.
- **Failure scenario:** Disk-full during `writeFileSync(tmpPath, ...)` at line 839 (after `entries` already includes the new change-log + 47 batch ops). Process crashes. Registry on disk is the pre-batch content (per rollback at lines 824-829). In-memory state is lost anyway. BUT: if a second process observes the registry between the rollback's `writeFileSync` and the second process's `readRegistry`, they see no envelope. If a third party retries the same batch, idempotency hash (Finding 3) matches — they rerun. Fine. But: the plan's "If the batch fails (any try/catch rollback path), NO change-log is emitted" claim is therefore not tested in its full breadth — only when an op inside the loop throws.
- **Evidence:** `core/meta-state.js:754-845` (full metaStateBatch body); the plan's Phase 2 step 9 "On rollback (any try/catch path), NO change-log is emitted" doesn't address rollback via writeFileSync/renameSync failures.
- **Suggested fix:** Specify the rollback contract precisely. Move the auto-emit to AFTER `renameSync(path)` succeeds — if renameSync throws, the registry wasn't updated and there's nothing to log. This requires restructuring the lock body to compute the envelope (pre/post snapshots, hash) BEFORE the file write, hold it in a local, and only push to entries+rewrite if the operation succeeded.

## Finding 8: Deny-list extension closes only the `meta_state_batch.update` path; write-on-change-log still allowed
- **Severity:** High
- **Location:** Plan § Architecture row "`operation_envelope` patchability today" + Phase 2 Architecture deny-list extension.
- **Flaw:** The plan claims `IMMUTABLE_PATCH_FIELDS` extension closes the batch hole. Verified true for `update` ops (meta-state.js:794-799 filters against the set). But the batch `write` op (line 773-777) does NOT consult the deny-list — `metaStateBatch` allows `op: "write"` with a fully-formed change-log entry including `operation_envelope`, bypassing the deny-list's purpose. The plan's claim "a `meta_state_batch.update` op with `operation_envelope: ...` is rejected with `immutable_field`" (Phase 2 requirement row) is true for update, but a write op with `entry: {...entry_kind: "change-log", operation_envelope: ...}` would still pass — Meta writes a new change-log entry with the envelope. That's an unlikely path, but the deny-list's intent ("no legitimate caller sets it via patch") is partially violated.
- **Failure scenario:** A buggy tool generates a write op with `entry.entry_kind === "change-log"` and `entry.operation_envelope === {...}`. The batch `write` op doesn't validate the deny-list. The change-log is written with an arbitrary envelope. Test (g) asserts `update` is rejected — passes. The `write` path is silently broken in the opposite direction. The plan's acceptance criterion ("a `meta_state_batch.update` op with `operation_envelope: ...` is rejected with `immutable_field`") doesn't mention this write gap.
- **Evidence:** `core/meta-state.js:773-777` (write branch has no deny-list check); `:794-799` (update branch does).
- **Suggested fix:** Add a corresponding check at `case "write"` that rejects change-log entries with `operation_envelope` if the source of that envelope wasn't the auto-emit path. Or document the write-path permissiveness explicitly in the plan and require tests to cover both branches.

## Finding 9: RED test (b) — `meta_state_log_change` will not throw SyntaxError; it returns `{logged: true}` with stripped field
- **Severity:** Medium
- **Location:** Phase 1, Step 4 (RED test (b)).
- **Flaw:** The plan predicts: "callTool throws SyntaxError from non-JSON MCP validation error." This assumes `.strict()` enforcement somewhere in the path. The MCP tool layer's schema (`metaStateChangeEntrySchema.pick(MIGRATED_FIELDS).shape`) is neither `.strict()` (Zod default is strip) nor wraps the input in a `try/catch` that re-throws as SyntaxError. The unknown-kind field will be stripped (the `kind` enum check inside survives if `operation_envelope` itself were re-added, see Finding 1), but the callTool succeeds. The test (b) "expect SyntaxError" assertion fails — not the assertion the plan provides. The plan's stated pattern ("same as Implementation 1") depends on the patch-tool's schema, which IS `.strict()`; this codebase's log-change tool is not.
- **Failure scenario:** Phase 1 RED run: test (b) calls `callTool("mastra_meta_state_log_change", {change_dimension, change_target, change_diff, reason, operation_envelope: {kind: "unknown-kind", ...}})`. Tool returns success with stripped envelope. Test asserts callTool throws — fails at the wrong assertion, not the registry-state check. The plan's fail-mode "registry state unchanged" still holds (registry state never changed, but for the wrong reason).
- **Evidence:** `meta-state-log-change-tool.js:31` (`schema: ...pick(MIGRATED_FIELDS).shape` — not `.strict()`); Implementation 1 patch tool at `meta-state-patch-tool.js:23` IS `.strict()` per the buildPatchSchemaFor note in `meta-state.js:343-346`.
- **Suggested fix:** Either add `.strict()` to the picked schema (and to `MIGRATED_FIELDS` — see Finding 1), OR rewrite test (b) to assert the post-call registry-state (which is unchanged because the field was stripped), without expecting a SyntaxError.

## Finding 10: Schema-extending Phase 1 may not break-or-fix existing tests as claimed
- **Severity:** Medium
- **Location:** Phase 1, Step 10 (Verify no regressions: `pnpm test:unit tools/learning-loop-mastra/core/meta-state.test.js`).
- **Flaw:** `metaStateChangeEntrySchema` is used by `meta_state_log_change` via `.pick(MIGRATED_FIELDS).shape` (Found 1) and is the union member of `metaStateEntrySchema` at `core/meta-state.js:263-271`. The union validator runs against the entire registry on `readRegistry`. Adding `operation_envelope: z.object({...}).optional()` to the change-log schema is strictly additive — but `metaStateEntrySchema` at line 263 uses `z.preprocess(withDefaults, ...)` which DEFAULTS `affected_system` to `"meta"`. If the new `operation_envelope`'s nested Zod schema ever has its own defaults or coercion, the preprocessor might apply them implicitly. Less likely: the new field uses `z.record(z.string(), z.number())` for `by_status`/`by_kind`, which captures ANY string key — so a registry entry with `operation_envelope: {by_status: {something_unexpected: "not a number"}}` would FAIL validation on read (parse failure). The plan doesn't spec validation-on-read behavior.
- **Failure scenario:** An auto-emit `buildEnvelope` produces `{by_status: {"active": 5, ...}}` (a legacy key, see Finding 5). The next `readRegistry` call validates this entry against `metaStateEntrySchema`. Zod rejects with coercion error. Tests in `core/meta-state.test.js` (which call `readRegistry`) fail with "validation_failed" errors. The "no regressions" verification fails.
- **Evidence:** `core/meta-state.js:263-271` (cross-cutting union validator); `:57-64` (`withDefaults` preprocessor); Phase 1 Zod snippet uses `z.record(z.string(), z.number().int().nonnegative())` — strict on values, open on keys.
- **Suggested fix:** Either close the `by_status`/`by_kind` keys via `z.enum([...canonical...])` (Finding 5 fix), or add a migration step in the plan to validate on read and emit a structured error rather than fail.
