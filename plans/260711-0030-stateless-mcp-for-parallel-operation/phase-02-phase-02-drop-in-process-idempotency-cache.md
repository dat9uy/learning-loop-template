# Phase 2: Drop in-process idempotency cache from log_change handler

## Overview

Remove the per-process 60s TTL idempotency cache (`_idempotencyCache`) in `tools/learning-loop-mastra/tools/handlers/meta-state-log-change-tool.js:10`. The cache was created to dedupe identical `(change_dimension, change_target, reason)` calls within 60s, but it created a **correctness surface**: the cache masks write failures by returning the cached `{logged: true, cache_hit: true}` even when the underlying writeEntry silently failed. Phase 1 added the cross-process lock so the write is actually safe; idempotency now belongs to the durable registry, not the per-process Map.

## Requirements

- **Functional**: Two `meta_state_log_change` calls within 60s with identical args must produce 2 distinct entries (each with a fresh `id` + `created_at`).
- **Non-functional**: Response shape remains backward-compatible except `cache_hit` field is removed.
- **Trust boundary**: Document explicitly that `meta_state_log_change` is a trusted-only tool. Add per-caller rate-limit at the transport layer (or document the gap with a follow-up plan). **(Finding 3)**

## Architecture

The cache was masking T1 (`meta-260606T2106Z`) and partially T4 (`meta-260619T2233Z`):
- T1: agent called log_change 5+ times in succession; cache returned same id, masking 4 silent failures
- T4: handler returned `logged: true` based on cached result, even when the underlying writeEntry silently dropped the entry

With Phase 1's cross-process lock, the underlying write is safe. Idempotency is now structural: the registry's `id` is generated by `generateId(slugify(change_target))` + `new Date().toISOString()` so two calls in the same millisecond produce different ids (one wins, the other gets a separate entry). The cache served no correctness purpose — only a perf optimization that crossed into correctness territory.

### Trust-boundary remediation (Finding 3)

Phase 2 removes the only intra-process rate-limit (`_idempotencyCache`). Without a replacement, any MCP-tool caller (including untrusted sub-agents) can flood `meta_state_log_change` at unbounded rate → registry disk-exhaustion, change-log pollution.

**Mitigation:** Document the trust boundary in `docs/runtime-contract.md` §3 (Record Routing — add a note: "All MCP-tool callers are trusted; the runtime is responsible for caller authentication at the transport layer."). File a follow-up loop-design for transport-layer caller authentication (unix-socket peer-credentials check or bearer-token). This follow-up lands separately to keep Phase 2 focused.

### Sibling `cache_hit` anti-pattern (Finding 15)

`meta-state-refresh-file-index-tool.js:39,99,119` has the same `cache_hit` anti-pattern. Phase 2 removes it from `log_change` but the sibling stays, leaving the silent-persistence-fail class half-closed.

**Mitigation:** Extend Phase 2 scope to also de-cache `meta-state-refresh-file-index-tool.js`. Update its description (line 39) to remove cache_hit mentions; remove the cache map (lines 99, 119, 145); keep the response field as `false` for backwards-compat (no consumers break). Add a regression test asserting 2 calls = 2 cache-misses (cache_hit: false always).

import { test } from "node:test";
import assert from "node:assert/strict";
import { mkdtempSync, rmSync, readFileSync } from "node:fs";
import { tmpdir } from "node:os";
import { join } from "node:path";
import { metaStateLogChangeTool } from "../../tools/handlers/meta-state-log-change-tool.js";
import { resolveRoot } from "../../tools/lib/resolve-root.js";

test("2 identical log_change calls within 60s produce 2 distinct entries", async () => {
  const root = mkdtempSync(join(tmpdir(), "drop-cache-test-"));
  const origRoot = process.env.GATE_ROOT;
  process.env.GATE_ROOT = root;
  try {
    // Initialize empty registry
    const fs = await import("node:fs");
    fs.writeFileSync(join(root, "meta-state.jsonl"), "", "utf8");

    const args = {
      change_dimension: "semantic",
      change_target: "tools/test/cache-drop",
      change_diff: {added: ["x"], removed: [], changed: []},
      reason: "RED test: cache-drop identical-args call",
    };

    // Fire 2 identical calls back-to-back
    const r1 = JSON.parse((await metaStateLogChangeTool.handler(args)).content[0].text);
    const r2 = JSON.parse((await metaStateLogChangeTool.handler(args)).content[0].text);

    // Assert: distinct entries, distinct ids
    assert.notEqual(r1.id, r2.id, "cache returned same id — drop failed");
    assert.equal(r1.logged, true);
    assert.equal(r2.logged, true);

    // Assert: registry has 2 entries
    const lines = readFileSync(join(root, "meta-state.jsonl"), "utf8").trim().split("\n").filter(Boolean);
    assert.equal(lines.length, 2, "expected 2 entries; got " + lines.length);

    // Assert: response shape no longer includes cache_hit
    assert.equal(r1.cache_hit, undefined, "cache_hit field should be removed");
    assert.equal(r2.cache_hit, undefined, "cache_hit field should be removed");
  } finally {
    process.env.GATE_ROOT = origRoot;
    rmSync(root, { recursive: true, force: true });
  }
});

test("registry is the source of truth: cache_hit field absent", async () => {
  // Already covered by test 1; this test asserts the response shape explicitly
});

## Risk Assessment

| Risk | Severity | Mitigation |
|---|---|---|
| Existing test breakage (cache_hit assertions) | Medium | Pre-scan with grep; targeted update of affected tests in Step 2.4 |
| Performance regression (no cache) | Low | Cross-process lock from Phase 1 is ~5ms; no per-call Map lookup saves 0.1ms; net ~+5ms per call, negligible |
| Two simultaneous identical calls produce 2 entries (semantic change) | Low | This is the **desired** behavior; user asked for it via Phase 1 lock. If the operator wants dedup, that's a future "duplicate detection" tool, not a silent cache |
| Agent reflex that relied on cache_hit to detect "I already logged this" | Low | Document in tool description; recommend `meta_state_list({id:...})` for post-write verification |
