## Finding 1: Phase 6 freshness check is a no-op — already implemented
- **Severity:** High
- **Location:** Phase 6, section "Phase 6.3 GREEN implementation" (`phase-06-phase-06-cross-process-cache-invalidation.md:156–158`) and "Phase 6 Architecture" (`:47–63`)
- **Flaw:** Phase 6 claims it adds sha256 freshness checking to the sidecar cache. It does not — the check is already there.
- **Failure scenario:** If Phase 6 is run "as planned", the only diff is a redundant parameter rename (`readLoopIntrospectCache(root, currentRegistrySha256)` → `readColdTierCache(root, cacheDir)`, both compute the same sha256 internally). No cross-process behavior changes. The plan's RED test `cross-process-cache-invalidation.test.cjs:115–119` calls `readLoopIntrospectCache(root, "sha256-of-empty-registry")` with a hard-coded sha — the existing implementation derives the sha itself, so the test signature mismatch means the test cannot even import the existing module.
- **Evidence:** `tools/learning-loop-mastra/core/loop-introspect-cache.js:31–45` already implements `readColdTierCache(root, cacheDir)` that computes `registrySha256(root)` (line 34) and compares against `cached.registry_sha256` (line 41), returning `{hit: false, reason: "sha_mismatch"}` (line 42). The plan's "Before" code (`:51`) and "After" code (`:52–63`) describe behavior that is already live.
- **Suggested fix:** Either skip Phase 6 entirely (the cross-process correctness it claims to add is already present), or rewrite Phase 6 to fix a real remaining gap. The plan should not be approved to ship a no-op that pretends to close a transport finding.

## Finding 3: Phase 5 nongit fallback collides across concurrent invocations
- **Severity:** High
- **Location:** Phase 5, "Architecture" `phase-05-phase-05-per-worktree-session-id.md:26–58`
- **Flaw:** Fallback `worktreeId = 'nongit-${process.pid}-${Math.floor(Date.now() / 1000)}'` (line 47) uses **wall-clock second truncation**. Two non-git processes started within the same second (e.g., a `node -e` and a `pnpm` invocation from `package.json` scripts), both with the same `process.pid` is impossible — but two **separate shells** within the same second using bash subshells can yield `process.pid` collisions only if the parent recycles PIDs, while the `Math.floor(Date.now() / 1000)` term is identical across both. The hash input becomes stable across multiple distinct sessions for ~1 second windows.
- **Failure scenario:** Operator runs `pnpm test` and `pnpm dev:web` in the same second in a non-git directory. Both shells' Claude Code sessions derive the same marker filename, so Session A's last-operator-message pollutes Session B's outbound-gate decision (the exact bug Phase 5 is supposed to close). The plan's "By design" risk note (line 191 of phase-05) attributes the same-worktree collision as intended, but does not anticipate cross-second cross-process non-git collision.
- **Evidence:** Plan line 47 of `phase-05-phase-05-per-worktree-session-id.md` shows the literal fallback template. No UUID, no nanosecond-precision timestamp, no random suffix. Comparison: git toplevel paths are inherently unique (sha256-prefix), but the fallback loses that property.
- **Suggested fix:** Add a random suffix (`crypto.randomBytes(4).toString("hex")`) or use the process start time in milliseconds. Or invert the fallback to *prefer* the git toplevel even when not in a git repo (use `init.defaultBranch` resolution + canonical abspath hash).

## Finding 5: Phase 4 schema-version-skew detection only checks the discriminator, not per-kind field drift
- **Severity:** High
- **Location:** Phase 4, "Architecture" `phase-04-phase-04-per-worktree-loop-version.md:74–102`
- **Flaw:** `isSchemaBranchSupported(root, entry.entry_kind)` only verifies that the writer's `entry_kind` is in the local `schema_branches` list (line 83). It does NOT verify that the writer's per-kind field shape (e.g., `loop-design.proposed_design_for`, `rule.applies_to_resolution`) is compatible with the reader's schema. Two consequences fall out:
  1. A v2.1 worktree with a `loop-design.proposed_design_for` array containing new entries silently stores them in the registry; a v2.0 reader sees them as `unknown shape` and either drops them (`metaStateEntrySchema` is a permissive union) or throws on `proposed_design_for: z.array(z.string()).min(1)` parse failure with no `entry_kind`-based guard.
  2. The lazy `.loop-version` generation (`:35–42`) reads from `tools/learning-loop-mastra/package.json` (the wrapper package, which doesn't exist per `ls tools/learning-loop-mastra/package.json` — see Bash error), so `readFileSync(join(root, "tools/learning-loop-mastra/package.json"), "utf8")` will throw.
- **Failure scenario:** A worktree running L2 v2.0 reads a registry entry written by L2 v2.1 with a new `rule.applies_to_resolution` field shape (e.g., a v2.1 multi-id list). The v2.0 schema `metaStateRuleEntrySchema` (line 189–222, `applies_to_resolution: z.string().optional()`) will reject the multi-id shape and throw `InvalidEntryError`, killing the read path (Phase 3's `readRegistry(root)` re-reads via Zod union at the time of `readRegistry` only — actually look at `_readAndParseRegistry` at `meta-state.js:372–385`, it does NOT call `metaStateEntrySchema.safeParse`, so the read tolerates the new shape silently). The write-side guard catches `entry_kind` mismatch but does NOT catch field-shape mismatch.
- **Evidence:** `tools/learning-loop-mastra/core/meta-state.js:189–222` shows `metaStateRuleEntrySchema.applies_to_resolution: z.string().optional()`. `meta-state.js:372–385` `_readAndParseRegistry` does NOT call `safeParse` on read — so reads are permissive, but writes go through `metaStateEntrySchema.safeParse` at line 537 (writeEntry path). Plan line 83–84: `if (entry.entry_kind && !isSchemaBranchSupported(root, entry.entry_kind)) throw...` — only discriminator. Plan `:46` references `tools/learning-loop-mastra/package.json` which does not exist (verified by `cat` exit 1).
- **Suggested fix:** Either narrow the scope claim to "entry_kind-discriminator skew only", or implement per-kind field-shape validation against a stored schema hash (like the lazy `.loop-version` hash + an `entry.preimage` declared in `metaStateEntrySchema`).

## Finding 6: Phase 1 lock-acquisition timeout vs. contention; 10s proper-lockfile default is below worst case
- **Severity:** Medium
- **Location:** Phase 1, "Architecture" `phase-01-phase-01-cross-process-file-lock.md:22–26` and "Risk Assessment" `:216`
- **Flaw:** Plan claims lock hold time "~5ms" and total throughput "≥10 writes/sec under 4 concurrent processes" (`:18`). The provided `withRegistryLock` retry config is `{retries: 5, minTimeout: 100, maxTimeout: 1000, factor: 1.5}` (line 156). Under 4-process contention where each write holds for 5ms, the 5-retry budget with exponential backoff can take ~100+150+225+337+500ms = ~1.3s before exhausting retries and timing out at the default 10s lock timeout. But the plan's stale window is only `stale: 5000` (line 156), so a 5+ second wait risks the lock being detected as stale and force-released mid-write, corrupting the registry.
- **Failure scenario:** With 4 concurrent MCP servers (worktree A, B, C, D each with their own process) all hammering `meta_state_log_change` for parallel PR operation (the plan's stated goal), the 5th retry of the slowest writer completes at ~1.3s elapsed, but partial writes from earlier retries may overlap with the next stale-detection sweep at 5s. The accepted-criteria lock-acquisition budget `<100ms in test environment` (`:209`) is asserted only for the 2-process test, NOT for the 4-process load the plan claims. Plan's "Lock contention under burst load" risk is rated "Low" with mitigation "5ms hold time" — that's true for 2 processes, false for 4.
- **Evidence:** Plan Phase 1 `:18` says "≥10 writes/sec under 4 concurrent processes" with no per-process timeout. `:156` is the actual code: `retries: 5, minTimeout: 100, maxTimeout: 1000, factor: 1.5` → worst-case cumulative wait = 100 + 150 + 225 + 337 + 500 = ~1.3s, with `stale: 5000` kicking in at 5s. Plan's stated risk is papered over. `core/meta-state.js:809–817` (explanation) acknowledges the per-process queue is the bottleneck and explicitly suggests `flock` — but doesn't quantify safe concurrency.
- **Suggested fix:** Add a stress test asserting 4-process lock acquisition completes within `stale: 5000` budget under 10 writes/sec sustained load; or lower `stale: 5000` to `stale: 2000` AND raise the default timeout to 30s; or back the lock with a synchronization-mode flag that detects live-holders (file content PID) before declaring stale.

## Finding 7: Bridge 5 "already shipped" is correct but the plan obscures that Bridge 5 ≠ registry locking
- **Severity:** Medium
- **Location:** Plan.md frontmatter `:17,47,88,179` and Phases 1–7 implication
- **Flaw:** Plan repeatedly asserts "Bridge 5 already shipped (status: completed in plan 260613-1853-phase-b-bridge-5-core-fix/plan.md)" — verified, frontmatter `status: completed` (line 4 of Bridge 5 plan). Bridge 5 closed T7 (the wire-format recursion bug) by deriving `meta_state_patch` schemas from `metaStateEntrySchema`'s 4 branches. **But Bridge 5's scope did NOT include a registry file lock.** The plan's framing implies Phase 1 is "natural continuation of Bridge 5's cross-process work" — when in fact Bridge 5 was strictly schema-derivation, and Phase 1 introduces a new dependency (`proper-lockfile`) and a new orchestration pattern that Bridge 5 has zero precedent for.
- **Failure scenario:** Operators reading the plan later (or PR reviewers) might assume Phase 1 inherits Bridge 5's battle-testing and reviews it more lightly. The plan's "Risks & Mitigations" row "Bridge 5 already-shipped confusion | Low" (`:169`) acknowledges confusion but doesn't flag the scope mismatch — Bridge 5 is unproven for the lock-acquisition path.
- **Evidence:** `plans/260613-1853-phase-b-bridge-5-core-fix/plan.md:32–40` describes Bridge 5 scope: "Derive the meta_state_patch tool's input schema from metaStateEntrySchema's 4 per-kind branches instead of the current z.object({}).passthrough()". Zero references to file locking in Bridge 5. `plans/260711-0030-stateless-mcp-for-parallel-operation/plan.md:88` table row "File locking primitive: `proper-lockfile`" attributes the dependency choice to "precedent: `tools/learning-loop-mastra/core/bound-artifacts.js:72`" — but `bound-artifacts.js:72` is a comment about "single-writer queue" in the context of `file-index.jsonl`, NOT about cross-process file locking. The precedent is for "single MCP-server writer" (per-process enqueue), not cross-process (`O_EXCL` style) locking. The dependency choice is novel for this codebase.
- **Suggested fix:** Decouple the Bridge 5 reference from Phase 1: re-label Phase 1 as "introduces proper-lockfile; no precedent in this codebase; cross-process locking is a new abstraction pattern that must be tested under sustained load."

## Finding 8: Phase 1 RED test uses `spawn(process.execPath, ...)` which DOES test cross-process, but the test infrastructure depends on shared module imports
- **Severity:** Medium
- **Location:** Phase 1, "Step 1.1 RED test" `phase-01-phase-01-cross-process-file-lock.md:60–127`
- **Flaw:** Each child imports `tools/learning-loop-mastra/core/meta-state.js` via absolute path (line 83). Each child V8 isolate re-evaluates the module on first import, so `writeQueues` (a module-level Map at `meta-state.js:355`) is fresh per process. That part correctly tests cross-process. BUT: the test creates the registry file via `fs.writeFileSync(registryPath, "", "utf8")` in the parent (line 76), and **does not initialize the .claude/coordination directory** that the lock file lives in (LOCK_PATH at `:150` is `.claude/coordination/.registry.lock`). The first writer that acquires the lock tries to write `lockPath`, and if the directory doesn't exist the lock acquisition will throw or fail.
- **Failure scenario:** The test never creates `<root>/.claude/coordination/`. The first child fails `properLock(lockPath, ...)` with ENOENT, the child's writeEntry rejects, the parent waits for `c.on("exit")` and rejects with `child exited non-zero`. Test fails spuriously.
- **Evidence:** Plan `:73–76` writes `meta-state.jsonl` to root but does not create the lock directory. Plan `:150` `LOCK_PATH = ".claude/coordination/.registry.lock"`. Plan `:154–158` `withRegistryLock` uses `properLock(lockPath, ...)` which expects the parent directory to exist (per `proper-lockfile` docs).
- **Suggested fix:** Add `mkdirSync(join(root, ".claude/coordination"), { recursive: true })` before spawning children, or change LOCK_PATH to live next to `meta-state.jsonl`.

## Summary

The plan's headline architecture is sound (cross-process file lock, drop idempotency cache, post-write visibility re-read, per-worktree versioning). Five of the seven phases are implementable as described, but **Phase 6 is a no-op against current code** (Findings 1+2), and **Phases 2, 4, 5 have concealed edge cases** (Findings 3–5). Phase 1 has a hidden test-fixture bug (Finding 8) and a contention claim that conflicts with its own retry config (Finding 6). The Bridge 5 cross-reference conflates scope (Finding 7).

Status: DONE_WITH_CONCERNS
Summary: Plan core architecture is implementable; two of seven phases (Phase 6 work, Phase 5 nongit fallback) need scope rewrites before approval; four edge cases need either guard code or scope narrowing.
Concerns/Blockers: Findings 1 and 2 collectively mean Phase 6 ships empty work unless the rewrite finds a real defect to fix. Findings 3 and 4 mean two phases will exhibit visible regressions in shipped product without code adjustments beyond what's specified.
