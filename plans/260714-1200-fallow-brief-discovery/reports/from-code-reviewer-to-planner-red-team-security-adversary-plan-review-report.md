## Finding 1: `meta_state_patch` fallback is impossible — `status` AND `consolidated_into` are both in `IMMUTABLE_PATCH_FIELDS`

- **Severity:** Critical
- **Location:** Phase 4, "Risk Assessment" section: "the fallback is `meta_state_patch` with `{ status: 'superseded', consolidated_into: '<change-log id>' }`"
- **Flaw:** The plan's documented fallback for the `meta_state_supersede` live-mode gate is itself blocked by the schema. A future operator trying to ship this plan in non-live mode will see `supersede` fail, attempt the `meta_state_patch` fallback, and have both `status` and `consolidated_into` rejected with `denied_by_IMMUTABLE_PATCH_FIELDS`.
- **Failure scenario:** Operator runs the plan in a dev environment (no `LOOP_SESSION_MODE=live`). `meta_state_supersede` returns `{superseded: false, reason: 'live_session_required'}`. Operator retries with `meta_state_patch({status: 'superseded', consolidated_into: '<change-log id>'})`. The patch path silently drops both fields, returning `{patched: true}` but the persisted entry still has `status: 'open'`, leaving the registry in an inconsistent state: the change-log claims to consolidate a finding that is still open. Downstream tools that read `isOpen(e)` will treat the finding as live; Rec 12 gap coverage will double-count or skip; Rec 10 dispatch handlers will fire on a "superseded" lineage ID. The fallback path is a documented road to a silently-corrupt registry.
- **Evidence:** `tools/learning-loop-mastra/core/meta-state.js:392-405` lists `status` (line 403) and `consolidated_into` (line 398) inside `IMMUTABLE_PATCH_FIELDS`. The deny-list enforcement is at `meta-state.js:1119-1125` (`denied = Object.keys(patch).filter((k) => IMMUTABLE_PATCH_FIELDS.has(k))`). The tool handler-comment at line 387-391 confirms the block is "stopgap until universal assertinvariant wrapper".
- **Suggested fix:** Drop the patch fallback. Either (a) make live-mode a precondition with no documented fallback, or (b) write a one-off `_admin/asset-internal` patch path that is gated by a separate capability flag and documented in the journal. The plan should also call `meta_state_resolve` (which is NOT in IMMUTABLE_PATCH_FIELDS — it goes through a dedicated path) only if the finding is meant to be `resolved` rather than `superseded`.

## Finding 2: Stale "60s idempotency cache" risk-mitigation references behavior that was DROPPED in plan 260711-0030

- **Severity:** High
- **Location:** Phase 2, "Risk Assessment" section: "`meta_state_promote_rule` has a 60s idempotency cache (per `meta-state-log-change-tool.js:9, 69-80` and verified in `plans/reports/journal-260628-fallow-tool-integration-rule.md` Lesson 6). If the same args are re-submitted within 60s, it silently no-ops."
- **Flaw:** The 60s cache was explicitly removed in plan 260711-0030. The plan's Phase 2 mitigation instructs implementers to "vary the `reason` parameter (or wait 60s) before retrying" for a cache that no longer exists in any of these tools.
- **Failure scenario:** Implementer reads the risk, hits a non-deterministic first-call failure, applies the "vary `reason`" workaround, then tries to revert. Meanwhile the actual fix (post-write `assertWriteVisible` re-read at `core/update-entry-helpers.js`, per `meta-state-log-change-tool.js:78`) is bypassed because the implementer trusts the stale text. Or worse: implementer believes the cache exists and never raises the real underlying issue (e.g., a file lock contention or a `live_session_required` rejection) because the symptom "silent no-op" reads as a cache hit rather than a real failure.
- **Evidence:** `tools/learning-loop-mastra/tools/handlers/meta-state-log-change-tool.js:7-11`: "Plan 260711-0030 Phase 2: in-process idempotency cache dropped. The previous 60s Map at this module masked silent-persistence-fail...". There is no 60s map in any of `meta-state-promote-rule-tool.js`, `meta-state-supersede-tool.js`, or `meta-state-log-change-tool.js`. The file:line citations in the plan (`meta-state-log-change-tool.js:9, 69-80`) point to non-cache code (the explanatory comment block and the schema-stripping lines).
- **Suggested fix:** Delete the risk-mitigation paragraph or rewrite it to cite `assertWriteVisible` (the actual idempotency mechanism post-260711-0030) and the `live_session_required` reason code. Update the journal cross-reference.

## Finding 3: H6 ordering-gate violation is silent during the Phase 2 implementation window — SessionStart hook strips the `warnings` array

- **Severity:** High
- **Location:** Phase 2, Implementation Steps 1 → 3-5 ordering (promote rule → append PROCESS_HINTS row); Phase 3, "Risk Assessment" (the parity-test path is described; the H6 race is not).
- **Flaw:** Phase 2 step 1 calls `meta_state_promote_rule` first (creates the consult-checklist rule entry), then step 3-5 adds the PROCESS_HINTS row. Between step 1 and step 5, the registry contains a `consult-checklist` rule whose `id` does not appear as a substring of any PROCESS_HINTS row. `loop_describe({tier:'warm'})` (line 94-106 of `loop-describe-tool.js`) computes `warnings = ["H6 ordering gate: consult-checklist rule ... has no corresponding PROCESS_HINTS row..."]`. The H6 warning is therefore emitted by the MCP server — but the SessionStart hook at `.factory/hooks/loop-surface-inject.cjs:281-312` (`formatBlock`) renders only `tool_count`, `rule_count`, `active_finding_count`, `LOCAL_DISCOVERABILITY_HINTS`, and `LOCAL_PROCESS_HINTS`. The `warnings` array is never serialized into the agent-facing block. The agent cannot see the violation through the documented discovery surface.
- **Failure scenario:** During the multi-commit rollout or a partial rebase, an implementer commits the registry rule before committing the PROCESS_HINTS row. The runtime claims `process_hints` "completeness" in the agent UI (tool_count is unchanged, hints render), but `loop_describe`'s warning array flags the cold-session. A future meta-state `sweep` audits this as a healthy rule entry that lacks a runtime surface — the registry is right, the runtime is wrong, and the agent has no way to learn about the gap. An attacker who can edit `meta-state.jsonl` but not `core/loop-introspect.js` (e.g., a PR that reorders commits) leaves the system in a state where the rule is enforced gate-side and the discovery-side hint never lands.
- **Evidence:** `tools/learning-loop-mastra/tools/handlers/loop-describe-tool.js:94-106` (H6 check), `.factory/hooks/loop-surface-inject.cjs:281-312` (renders 5 fields, never `warnings`), `tools/learning-loop-mastra/__tests__/legacy-mcp/loop-describe-tool.test.js` (no test for the warnings-array-as-user-visible content).
- **Suggested fix:** Either (a) reverse the Phase 2 ordering: edit `core/loop-introspect.js` BEFORE promoting the rule, or (b) add a one-time `meta_state_log_change` entry that explicitly cites "H6 ordering gate pre-curtain" so the registry reflects the implementation window. Either way the plan should call this out.

## Finding 4: `change_target` "200-char limit" is a different field — the risk-mitigation is based on a misread of the schema

- **Severity:** Medium
- **Location:** Phase 4, "Risk Assessment": "The change-log's `change_target` string is too long (limit is 200 chars per the schema). Mitigation: shorten to `'meta-state.jsonl#rule-fallow-brief-on-gate-failure'`"
- **Flaw:** The plan cites a 200-char limit that exists in the schema but applies to a DIFFERENT field (`operation_envelope.target`, `core/meta-state.js:253`), not `change_target` (`core/meta-state.js:208-209`, only `.min(1)`). The plan's compound `change_target` in Phase 4 step 2 is 244 chars and will validate fine; the mitigation is unnecessary AND would DESTRUCTIVELY shrink the audit trail to a single token — losing the `core/loop-introspect.js` and `.factory/hooks/loop-surface-inject.cjs` references that are the reason for the change-log entry's existence.
- **Failure scenario:** Implementer reads the risk, applies the mitigation, shortens `change_target` to a single token. Rec 12 gap-detection (`tools/learning-loop-mastra/core/change-log-bound-paths.js:226-232`, the `canonicalizeChangeTarget` set used by `findGapCandidates` at `loop-introspect.js:331`) now only covers `meta-state.jsonl`. Edits to `tools/learning-loop-mastra/core/loop-introspect.js` and `.factory/hooks/loop-surface-inject.cjs` no longer get coverage from this entry — the gap detector will surface them as advisory gaps on the next session start, even though the change-log nominally records them. The audit trail is degraded precisely for the files that were the point of the plan.
- **Evidence:** `tools/learning-loop-mastra/core/meta-state.js:208` (`change_target: z.string().min(1)`) vs line 253 (`target: z.string().min(1).max(200)` — the `operation_envelope.target` field). Compare against the existing change-log at `meta-state.jsonl` line 181-201 (the canonical model) whose `change_target: "tools/learning-loop-mastra/tools/handlers/meta-state-sweep-tool.js"` is 60 chars but analogous entries use much longer compounds per the `splitCompound` delimiter (`+`) at `change-log-bound-paths.js:184-186`.
- **Suggested fix:** Remove the risk-mitigation paragraph. The 244-char compound is canonical and will validate; `splitCompound` will canonicalize it into 3 repo-relative paths.

## Finding 5: `.factory/hooks/loop-surface-inject.cjs` is NOT in `CHANGE_LOG_BOUND_PATHS` — gap-detection will silently de-prioritize drift on the hook file

- **Severity:** Medium
- **Location:** Phase 1, step 4 ("Read .factory/hooks/loop-surface-inject.cjs LOCAL_PROCESS_HINTS to confirm it currently has 4 rows"); Phase 2 step 5 ("mirror row #5 byte-for-byte"); Phase 4 step 2 (compound `change_target` includes `.factory/hooks/loop-surface-inject.cjs#LOCAL_PROCESS_HINTS row #5`).
- **Flaw:** The plan puts `.factory/hooks/loop-surface-inject.cjs` in the change-log `change_target`, but the Rec 12 gap-detector allowlist at `tools/learning-loop-mastra/core/change-log-bound-paths.js:41-52` includes `.factory/skills/**`, NOT `.factory/hooks/**`. The canonicalizer (`change-log-bound-paths.js:226-232`) will preserve the path but `isBoundPath` at line 85-88 returns `false` for it. The gap-builder (`findGapCandidates` at `tools/learning-loop-mastra/core/loop-introspect.js:331-337`) will ignore edits to the hook file entirely — even when the change-log entry's `change_target` literally lists the path.
- **Failure scenario:** A future PR edits `.factory/hooks/loop-surface-inject.cjs` to drift the LOCAL_PROCESS_HINTS mirror. The cold-session parity test (`tools/learning-loop-mastra/__tests__/legacy-mcp/cold-session-discoverability.test.cjs:341-356`) catches the byte-level drift immediately. But the broader Rec 12 advisory will not surface this drift in `gap_candidates` because the path is excluded by `isBoundPath`. The Rec 12 advisory narrows the operator's attention away from exactly the file the plan modifies — the asymmetry is documented in the change-log itself (the path is dropped silently because the gap-detector doesn't consider it "bound"). This is the second time the plan's surface area is wider than the registry recognizes.
- **Evidence:** `tools/learning-loop-mastra/core/change-log-bound-paths.js:41-52` (allowlist), `tools/learning-loop-mastra/core/loop-introspect.js:331-337` (`findGapCandidates` applies `isBoundPath`), `.factory/hooks/loop-surface-inject.cjs:1` (the file exists at the path).
- **Suggested fix:** Either (a) add `.factory/hooks/**` to `CHANGE_LOG_BOUND_PATHS` in this plan, or (b) explicitly note in the journal that the `.factory/hooks/` surface is intentionally outside the bound-paths detection and document where it IS audited (the parity test, only).

## Finding 6: Compound `change_target` loses `#anchor` precision in the canonicalizer — audit-trail granularity is degraded

- **Severity:** Medium
- **Location:** Phase 4, Implementation Step 2: `change_target: 'meta-state.jsonl#rule-fallow-brief-on-gate-failure + tools/learning-loop-mastra/core/loop-introspect.js#PROCESS_HINTS row #5 + .factory/hooks/loop-surface-inject.cjs#LOCAL_PROCESS_HINTS row #5'`
- **Flaw:** The canonicalizer at `core/change-log-bound-paths.js:98-101` (`stripAnchor`) drops every `#`-suffixed qualifier. After canonicalization, the Rec 12 gap-detector sees ONLY the three repo-relative paths; the `row #5` / `rule-...` qualifiers that distinguish "added row #5" from "rewrote row #1" are gone. A future audit that asks "did the change-log entry correctly identify which PROCESS_HINTS row was edited" cannot answer from the canonicalized set alone.
- **Failure scenario:** Operator queries `meta_state_change_log_gaps` and sees `{ gap_candidates: ['tools/learning-loop-mastra/core/loop-introspect.js'], gap_protocol_prompt: '...' }`. The operator cannot distinguish "this entry covers edits to row #5 only" from "this entry covers a full rewrite of the file" — both reduce to the same prefix. If a future adversarial or accidental edit touches row #1 of `loop-introspect.js` without a new change-log, the gap detector silently considers it covered by the entry from this plan.
- **Evidence:** `tools/learning-loop-mastra/core/change-log-bound-paths.js:98-101` (anchor stripping is unconditional — every token with `#` loses the suffix). The asymmetry is documented at `change-log-bound-paths.js:99` as "Anchors are not paths; they name functions/symbols/keys within the file." (so it is by-design, but the plan does not acknowledge the audit-trail loss). The single-token mitigation in Finding 4 would PRESERVE `meta-state.jsonl#rule-fallow-brief-on-gate-failure` only because the existing canonicalizer doesn't distinguish "rule entry" from "rewrote the whole file".
- **Suggested fix:** Use the change-log's `evidence_code_ref` field (single path per entry, no anchor stripping in the canonicalizer — see `change-log-bound-paths.js:140-148` which preserves bare repo-relative paths) for the precise file pointer, and keep the compound `change_target` as the human-readable audit trail that does not feed the canonicalizer.

## Finding 7: `pattern` field is `z.string()`, not parsed JSON — a poisoned body can survive the schema gate

- **Severity:** Medium
- **Location:** Phase 2, Implementation Step 1 (the `pattern` parameter — Appendix A's 1-item checklist JSON); plan "Architecture" section, Appendix A
- **Flaw:** The rule body's `pattern` is typed `z.string()` (`core/meta-state.js:282`) — it is stored as a JSON-encoded string but the schema does NOT enforce valid JSON. The promote-rule tool at line 162-176 writes whatever string `pattern` is supplied. A malformed or adversarially-framed JSON payload would be persisted verbatim, and consumers that `JSON.parse(rule.pattern)` (e.g., the existing test at `tools/learning-loop-mastra/__tests__/legacy-mcp/gate-logic-consult-checklist-tool-integration.test.js:25` and presumably the runtime `applyPromotedRules`) would silently throw at read time.
- **Failure scenario:** A PR adds `pattern: 'definitely-not-json'` to a future rule (or, in this plan's case, an editor accidentally submits `pattern: '<JSON-with-trailing-comma>'`). The promote tool writes the entry. `loadPromotedRules` returns the entry. The cold-session parity test passes (it only checks `PROCESS_HINTS` substring match). At session start, `loop_describe({tier:'warm'})` builds `process_hints` and a downstream consumer hits `JSON.parse(rule.pattern)`, throwing inside a non-recoverable path. The hook at `.factory/hooks/loop-surface-inject.cjs:281` has a `formatBlock` that does not consume `rule.pattern`, so its swallow-the-error contract is fine for that surface, but the upstream MCP server can crash, killing every cold-session agent.
- **Evidence:** `tools/learning-loop-mastra/core/meta-state.js:282` (`pattern: z.string()` — no `z.string().refine(...)` JSON guard), `tools/learning-loop-mastra/tools/handlers/meta-state-promote-rule-tool.js:162-176` (writes the raw string into `ruleEntry.pattern` with no `JSON.parse` validation).
- **Suggested fix:** The plan itself is safe (Appendix A is well-formed JSON), but the plan should NOT ship without an upstream JSON-validation guard on the `pattern` field of any `consult-checklist`-typed rule. Add `try { JSON.parse(pattern) } catch { return reject }` to the `meta_state_promote_rule` tool OR to `loadPromotedRules`'s consult-checklist branch. Recommend filing this as a finding-entry rather than fixing in this plan.
