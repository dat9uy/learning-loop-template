### Phase 1: Pure Function `queryDrift` (TDD, 24 Unit Tests)

Wrote 24 stubbed tests first, confirmed RED (`ERR_MODULE_NOT_FOUND`). Created the pure function with:
- Source-of-truth join logic: `computeIsDrift(derivation, grounding, entry)` enumerates 6 cases (resolved, active-uncertain, code-missing, SP2-drifted, etc.)
- `computeRecommendation(derivation, grounding)` returns `"resolve"` or `"investigate"`
- 4-case join table maps (SP1 status, SP2 status) → drift + recommendation
- Change-log fast path: `derivation.kind === "no-signals"` early-returns
- Filter-agnostic: function takes already-filtered entries; the tool layer does the filtering

All 24 unit tests pass. Regression-safety: 621 + 24 = 645.

**Deviation from plan during implementation (1):**
- The plan's `computeIsDrift` only listed 3 cases (resolved, code-missing, SP2-drifted). I added the `active-uncertain` case because the test T-3 requires drift for `code-only` (active-uncertain) entries. The plan's `computeRecommendation` already had the `active-uncertain → investigate` branch; only the `computeIsDrift` was missing the case. This is a fix to the plan, not a deviation from intent.

### Phase 2: `meta_state_query_drift` Tool (TDD, 24 Tool Tests)

Wrote 24 stubbed tests first, confirmed RED. Created the tool with:
- Imports: `readRegistry`, `filterEntries` from `core/meta-state.js`; `queryDrift` from `core/query-drift.js`; `resolveRoot` from `lib/resolve-root.js`; `appendGateLog` from `lib/gate-logging.js`
- Schema: Zod (per SP1/SP2 pattern, NOT raw shape — see deviation below)
- `resolveRoot` wrapped in try/catch (returns `context_load_failed` on error)
- Builds `codeContext = { root, run_grounding, run_tests: false, test_passed: null, now: () => Date.now() }`
- Filters the registry by `filter?.status` (active or reported); passes non-terminal entries to `queryDrift`
- Appends 1 gate log line per call: `{ event: "meta_state_query_drift", filter, run_grounding, drift_count }`

All 24 tool tests pass. Regression-safety: 645 + 24 = 669.

**Deviation from plan during implementation (1, significant):**
- The plan specified a raw JSON-Schema-style `inputSchema` object. The MCP SDK 1.29.0 requires a Zod schema (or `ToolAnnotations`). The gate-integration test caught this: the server refused to register the tool. I rewrote the schema to use Zod, mirroring the SP1/SP2 sibling tools. The behavior is identical to the spec; only the schema representation changed.

## Key Decisions

1. **Added case 5 (active-uncertain) to `computeIsDrift`:** the plan's implementation was missing this case. Without it, `code-only` entries (code exists, test missing) wouldn't surface as drift. The plan's `computeRecommendation` already had the `active-uncertain → investigate` branch, so the intent was clear; only the drift detection was missing. Fixed in Phase 1.

2. **Tool schema converted from raw shape to Zod:** the plan specified `inputSchema: { type: "object", properties: { ... } }` (raw JSON-Schema style). The MCP SDK 1.29.0 rejected this with `"expected a Zod schema or ToolAnnotations, but received an unrecognized object"`. Rewrote the schema to use Zod, matching the SP1/SP2 sibling tools. The wire behavior is unchanged. Documented in the deviation log.

3. **Acceptance tests mirror SP2's pattern:** used the same `mkdtempSync` + `process.env.GATE_ROOT` test pattern as `sp1-derive-status-acceptance.test.js` and `sp2-check-grounding-acceptance.test.js`. The real finding is copied to a temp registry; the referenced file is recreated in the temp dir to control SP1's output.

4. **Cook journal written before Phase 4 (at end of Phase 3) per SP2 cook pattern:** the SP2 cook journal was also written before the docs-update phase. The journal includes a Phase 4 addendum.
