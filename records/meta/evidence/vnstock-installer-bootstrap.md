# vnstock_data Installer Bootstrap Discovery

This is loop-architecture evidence for the broken-assumption discovery surfaced during phase 04 of `decision-20260510T160000Z-capabilities-stack-migration`, and the proposed two-stage bootstrap path forward.

## Prior Decision Tradeoff (Broken Assumption)

`records/decisions/decision-20260510T160000Z-capabilities-stack-migration.yaml` carried the tradeoff:

> Recreating `product/api/.venv` (rather than `mv`-ing the existing `.venv`) requires running `uv sync` against PyPI. The vnstock_data wheel is on PyPI (`INSTALLER: uv` per current dist-info), so this works without re-triggering the Makeself installer or vendor device-clearance flow.

The `INSTALLER: uv` field in dist-info indicates the local tool that performed the install, not the registry the artifact came from. No `vnstock_data==3.1.7` wheel exists on PyPI or in any registry uv can resolve from default sources.

## Phase 04 Observation

Per `records/evidence/loop/capabilities-stack-migration.md`:

> `uv sync --extra vendor` could not resolve the private `vnstock_data` distribution from the current registry. The stack virtualenv was recreated, public dependencies were installed from the package index, and private vendor packages were copied from the old local virtualenv after the registry miss.

The migration shipped, but the resulting `product/api/.venv` is reproducible only via the hand-copy step. A fresh clone, a CI run, or any operator deleting `product/api/.venv` cannot rebuild the venv from the repo alone.

## Root Cause

Per `records/claims/claim-vnstock-install-sandbox.yaml` and `records/evidence/vnstock-data/installer-prior-notes.md`:

- Vendor distributes `vnstock_data` as a Makeself self-extracting `.run` archive at `https://vnstocks.com/files/vnstock-cli-installer.run`.
- The installer creates its own venv (observed at `$HOME/.venv` per sandbox-1), registers the executing device against the vendor account (consumes one tier slot), then downloads + extracts `vnstock_data` (and `vnstock_core`) into that venv.
- The artifact uv would need (a wheel) does not exist as a vendor-published artifact. The dist-info inside the installed `.venv` is generated by the installer's internal copy, not by a wheel resolve.

Per `records/claims/claim-vnstock-device-limit-mechanism.yaml`, each fresh install consumes one Linux device slot under the bronze-tier subscription state. Operator handles slot accounting via the vnstocks UI per `decision-20260509T070411Z-vnstock-vendor-device-limit-clearance`.

Per `records/evidence/vnstock-data/experiment-install-20260509T071800Z-sandbox-1.md`, the verified install:

- Installer SHA-256: `1982f7f93386daa57e1a1c0b18a87c8a299b9ad1d331f4123ab534656fa4c7ed`.
- Substrate prerequisites: working venv with `requests`, `uv`, and `pandas` already importable. Without `pandas`, the installer fails before vendor authentication during `vnai`/`vnii` verification.
- Env-var-driven: `VNSTOCK_API_KEY`, `VNSTOCK_CONFIG_PATH`, `VNSTOCK_VENV_TYPE=venv`, `VNSTOCK_LANGUAGE=python`. `VNSTOCK_VENV_PATH` is not honored — installer always lands in `$HOME/.venv`.

## Workaround in Effect

Today's `product/api/.venv`:

- Public deps (`pandas`, `requests`, `uv`) installed via `uv sync` from the public package index.
- Private vendor packages (`vnstock_data`, `vnstock_core`) hand-copied from a pre-migration `.venv`.
- `product/api/pyproject.toml` declares an `[project.optional-dependencies] vendor` extra pinning `vnstock_data==3.1.7` — a resolution promise uv cannot keep.

## Reproducibility Gap

Today's state:

- A fresh clone cannot run `uv sync --extra vendor` and obtain a working venv.
- CI cannot rebuild the venv without bringing the hand-copied package files into the build path.
- The `vendor` extra in `pyproject.toml` is misleading config — it advertises a resolution path uv cannot satisfy.

## Proposed Path Forward

Two-stage bootstrap, scoped in `plans/reports/brainstorm-260510-1706-vnstock-installer-bootstrap.md`:

- Stage 1 — `uv sync` for public deps only. Drop the `vendor` extra from `pyproject.toml`.
- Stage 2 — `product/api/scripts/install-vnstock.sh` downloads the Makeself installer, verifies SHA against the value pinned above, executes it with `HOME="$(realpath product/api)"` so its `$HOME/.venv` resolves to the same venv stage 1 created. Idempotent: pre-flight `import vnstock_data` short-circuits a successful re-run.

Records to author during the implementation plan:

- New evidence MD documenting a fresh `pnpm bootstrap:api` run on a clean `.venv`.
- Decision flip from draft to approved (`decision-20260510T170623Z-vnstock-installer-bootstrap`).
- Hash-pin rotation event recorded as a sandbox-1-style experiment whenever the vendor publishes a new installer.
